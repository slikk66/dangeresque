## Summary

Hypothesis in the issue is **correct**. `getWorktreeResults` reads the archive
directory relative to `projectRoot` instead of the worktree path, so pre-merge
runs always show "No run artifacts" even when the `.md`/`.json` pair sits on the
worktree branch.

### Root cause — exact lines

`src/worktree.ts:340-407` `getWorktreeResults`:

```
line 372:    const archived = listArchivedRuns(projectRoot, issueNum);
line 377:    const content = readArchivedRun(projectRoot, issueNum, archived[i]);
line 383:    const latest  = readArchivedRun(projectRoot, issueNum, latestName);
```

Both helpers join their first arg with `.dangeresque/runs/issue-<N>/`
(`getRunsDir` at `src/worktree.ts:149-151`, `getIssueRunsDir` at 153-155).
Passing `projectRoot` points the reader at
`<projectRoot>/.dangeresque/runs/issue-<N>/` — which is only populated **after**
`dangeresque merge` lands the worktree's commits on main.

Actual artifact location is `<worktree>/.dangeresque/runs/issue-<N>/…`, set by
`computeRunArchivePath` at `src/runner.ts:42-50`:

```ts
return join(worktreePath, CONFIG_DIR, RUNS_DIR, subdir, `${timestamp}-${mode}.md`);
```

and committed to the worktree branch by `commitArchiveFile` at
`src/runner.ts:52-61`. The reader in `getWorktreeResults` was not updated to
match when the migration happened.

### Provenance

Introduced by commit `a818787` — "track run artifacts in git; fail loudly; no
worktree reuse" (2026-04-20). That commit moved artifacts from a worktree-root
`RUN_RESULT.md` into `<worktree>/.dangeresque/runs/issue-<N>/…` so they flow
through `git merge`, but left `getWorktreeResults` reading from `projectRoot`.
The same commit added a stale docstring at `src/worktree.ts:145-148`:

```
// Workers write their run result directly to .dangeresque/runs/issue-<N>/<timestamp>-<MODE>.md
// (in the project root, not the worktree). These helpers read that directory.
```

which contradicts the code in `runner.ts` written in the very same commit. The
comment is wrong; the runner is right.

### Why `results --issue <N>` still works for merged issues

`cmdResults` (src/cli.ts:580-613) dispatches `--issue <N>` to
`getArchivedResults` (src/worktree.ts:412-445), which is intended for post-merge
lookup and correctly roots at `projectRoot` — files are on main at that point.
Only the active-worktree path (`--latest` / `<branch>` → `getWorktreeResults`)
is buggy. `cleanArchivedRuns` (207-217) also correctly uses `projectRoot` —
`dangeresque clean --issue N` deletes the post-merge history, not the worktree
copy.

### Reproduction (confirmed by code inspection)

1. `dangeresque run --issue 14` writes artifact to
   `.claude/worktrees/dangeresque-investigate-14/.dangeresque/runs/issue-14/<ts>-<MODE>.md`
   and commits it on branch `worktree-dangeresque-investigate-14`.
2. `dangeresque results --latest` picks this worktree (most-recent commit),
   extracts issue 14 from branch name, then asks
   `listArchivedRuns(projectRoot, 14)` → reads
   `<projectRoot>/.dangeresque/runs/issue-14/`.
3. Because nothing is merged yet, that directory is empty (or absent) — prints
   `No run artifacts in .dangeresque/runs/issue-14/`.
4. The following diff-summary block runs in `targetWorktree.path`, so it
   correctly lists the `.md` + `.json` pair — contradicting the "no run
   artifacts" message directly above it. That mismatch is exactly what the
   issue describes.

I verified the directory layout in this very worktree:
`/…/dangeresque-investigate-14/.dangeresque/runs/issue-12/` contains the merged
artifacts from a prior run (proof the structure is tracked in git and carries
across worktrees). `issue-14/` is empty at the time of this write because the
current run's artifact (this file) lands only after my session ends.

## Proposed Fix (not applied — INVESTIGATE mode)

Single-file change in `src/worktree.ts`, inside `getWorktreeResults`:

```
-    const archived = listArchivedRuns(projectRoot, issueNum);
+    const archived = listArchivedRuns(targetWorktree.path, issueNum);
 …
-          const content = readArchivedRun(projectRoot, issueNum, archived[i]);
+          const content = readArchivedRun(targetWorktree.path, issueNum, archived[i]);
 …
-      const latest = readArchivedRun(projectRoot, issueNum, latestName);
+      const latest = readArchivedRun(targetWorktree.path, issueNum, latestName);
 …
-      lines.push(`No run artifacts in .dangeresque/runs/issue-${issueNum}/`);
+      lines.push(`No run artifacts in ${targetWorktree.path}/.dangeresque/runs/issue-${issueNum}/`);
```

Also correct the stale docstring at `src/worktree.ts:145-148` so it matches
reality ("Workers write their run result inside the worktree at
`<worktree>/.dangeresque/runs/issue-<N>/…`. Some helpers below read the
post-merge copy at the project root; see each function's docstring.").

Leave `getArchivedResults` and `cleanArchivedRuns` alone — they are correct for
their post-merge use case and the issue explicitly requires unchanged behavior
there.

No `projectRoot` use is removed from `getWorktreeResults` — `listWorktrees`
still needs it, and the diff-summary already uses `targetWorktree.path`. The
function signature does not change.

## Observations

- The misleading `// (in the project root, not the worktree)` comment at
  `src/worktree.ts:145-148` is likely what caused the bug during the `a818787`
  migration — the author/tool updated one side of the split but read the wrong
  intent from the old docstring.
- The "No run artifacts in `.dangeresque/runs/issue-<N>/`" message at
  `src/worktree.ts:387` hard-codes a relative path in the error text, which is
  why the issue reporter could not see where the lookup was actually pointing.
  Including the absolute path in the error text (see proposed fix) will
  eliminate this class of confusion in the future.

## Risks / Uncertainty

- **None identified.** The fix is a narrow parameter swap with no change to
  public API, artifact layout, or merge semantics. `getArchivedResults` /
  `cleanArchivedRuns` remain untouched, so `--issue N` behavior on merged
  issues is guaranteed unchanged by construction (literally no edits to those
  functions).
- Worktree-relative artifacts are already committed on the worktree branch at
  the time `results` is invoked (commit happens inside `commitArchiveFile`
  immediately after the worker exits 0, before the command returns), so
  reading the worktree working tree will always see the committed file.
