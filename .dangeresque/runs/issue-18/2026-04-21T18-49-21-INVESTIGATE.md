## Summary

Dangeresque has four implicit-default "footguns" that the issue asks to remove:

1. `dangeresque logs` (no args) → picks latest worktree silently
2. `dangeresque results` (no args) → picks latest worktree silently
3. `dangeresque results --latest` → explicit latest flag (still a footgun: "latest of what?")
4. `dangeresque run` with neither `--issue` nor `--issue-fixture` → silently reads `NEXT_TASK.md`

Every implicit path + its supporting code (TASK_FILE constant, ADHOC_DIR, `NEXT_TASK.md` branch in `buildTaskPrompt`, `getWorktreeResults("latest")` branch, `--latest` help/docs) is dead after this change and should be fully excised.

Confidence: **HIGH**. The scope is well-bounded (one-line issues into the CLI + a small amount of dead-code removal) and every call site is traceable statically. No runtime behavior outside these commands is affected.

Extract a helper used by both `cmdLogs` and `cmdResults`:

```ts
function formatMissingTargetError(cmd: string, argName: string, worktrees: WorktreeInfo[]): string {
  let msg = `Usage: dangeresque ${cmd} ${argName} [options]\n\n`;
  if (worktrees.length === 0) {
    msg += "No active dangeresque worktrees.";
    return msg;
  }
  msg += "Active worktrees:\n";
  for (const wt of worktrees) {
    const shortBranch = wt.branch.replace(/^worktree-dangeresque-/, "");
    let state = "IDLE";
    if (wt.running && wt.pidInfo) {
      const elapsed = formatElapsed(Date.now() - wt.pidInfo.startedAt);
      state = `RUNNING (pid ${wt.pidInfo.pid}, ${elapsed} elapsed)`;
    } else if (wt.pidInfo && !wt.running) {
      state = "IDLE (worker exited)";
    }
    msg += `  ${shortBranch.padEnd(20)} ${state}\n`;
  }
  msg += `\nPass one explicitly, e.g.: dangeresque ${cmd} ${worktrees[0].branch.replace(/^worktree-dangeresque-/, "")}`;
  return msg;
}
```

## Risks / Uncertainty

1. **`getWorktreeResults` signature change is a public API break** (re-exported from `src/index.ts`). External TS consumers passing `"latest"` will stop compiling. No known external consumers, but worth noting in the commit message.
2. **`TASK_FILE` export from `src/index.ts`** — same concern. Removing from public API. No known consumers.
3. **Short-branch name in `formatMissingTargetError`** — the example in the issue uses shorthand like `investigate-12`, but `listWorktrees` returns full branch names (e.g. `worktree-dangeresque-investigate-12`). The IMPLEMENT plan strips the `worktree-dangeresque-` prefix. This matches existing behavior in `resolveBranch` (accepts shorthand) and `cmdStatus` (prints full branch). If the issue author wants the full branch printed, that's a one-line tweak.
4. **Known-flag whitelist is a small new pattern**. Currently the CLI doesn't do strict flag validation anywhere. Adding it only to `cmdLogs` and `cmdResults` (the two commands where `--latest` needs to become an error) is consistent with the issue's scope and avoids widening.
5. **Fixture-only runs** — `cmdRun --issue-fixture <path>` sets `issueNumber = issueData.number` (cli.ts:206). So `issueNumber` is always defined before `runWorker` is called. `computeRunArchivePath(_, issueNumber, _)` with required `issueNumber` is sound.
6. **Prior run artifacts under `.dangeresque/runs/issue-11/…12/…14/…`** contain historical references to `NEXT_TASK.md` and `adhoc/`. These are immutable records of past work. Per AFK_WORKER_RULES scope guidance ("do not modify files you did not create in this run"), the IMPLEMENT run should NOT touch them. The grep assertions above explicitly exclude this directory.
7. **Shared helper location** — `formatElapsed` is already in `cli.ts`. The new `formatMissingTargetError` also belongs there (not a candidate for extraction to another module unless the CLI grows more). `WorktreeInfo` is already imported via `listWorktrees`, so no new import needed.
