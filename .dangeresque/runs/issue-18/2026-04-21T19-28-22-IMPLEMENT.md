## Summary

Removed every implicit-default path from the CLI and every line of code supporting those paths. Each of the four footguns in the issue is now an explicit error with a helpful usage hint:

1. `dangeresque logs` (no positional) → prints `Usage: dangeresque logs <branch> [options]` + active-worktree list (or "No active dangeresque worktrees.") + `process.exit(1)`. `--latest` now falls into the unknown-flag path and errors.
2. `dangeresque results` (no positional, no `--issue`) → same helpful error. `--latest` → unknown flag error. `--issue <N>` path is untouched.
3. `dangeresque run` (no `--issue` / `--issue-fixture`) → prints "A task source is required" + the two valid invocations + `process.exit(1)`. `NEXT_TASK.md` is never read.
4. `NEXT_TASK.md` / `TASK_FILE` / `ADHOC_DIR` / `adhoc/` — all supporting code deleted (constants, imports, unreachable branches, prompt-template language).

The `getWorktreeResults` signature narrowed from `(root, branchOrLatest: string | "latest")` to `(root, branch: string)`. `RunOptions.issueData` changed from optional to required. `computeRunArchivePath` `issueNumber` changed from `number | undefined` to `number` (and the `ADHOC_DIR` branch removed). `buildTaskPrompt` collapsed to the issue-driven branch only. `buildReviewPrompt` lost its `opts.issueData ? ... : ...` ternary since issueData is always present.

## Risks / Uncertainty

1. **`yarn build` not exercised** — this is the biggest gap. I verified type correctness by reading the diff and tracing call sites; the analysis is below. A VERIFY run should run `yarn build` as its first check.
   - `let issueData: IssueData;` with no initializer plus try/catch-calls-exit pattern relies on `process.exit` being typed `never` (which it is in `@types/node`). TS 6.x definite-assignment analysis should handle this.
   - `issueNumber!` non-null assertion is sound because the two guards together (mutex + required-source) prove exactly one of the two is defined, and the `else` branch runs when `issueFixturePath === undefined`, hence `issueNumber !== undefined`.
   - The `WorktreeInfo` and `IssueData` type imports exist in their source files and are type-only imports, so no runtime impact.
2. **Public API narrowing**:
   - `getWorktreeResults` — external TS consumers passing `"latest"` will break at compile time. This is the intended breaking change.
   - `TASK_FILE` — removed from `src/index.ts` export list. External consumers importing it will fail to compile.
3. **Edge case: `--issue 0`** — pre-change, `else if (issueNumber)` treated 0 as falsy and silently skipped the fetch. Post-change, the `else` branch unconditionally runs and `fetchIssue(_, 0)` produces a clear `gh` error ("Failed to fetch issue #0 …"). Behavior is slightly different but more correct — 0 is not a valid issue number anyway.
4. **`dangeresque results --issue 63 --latest`** — `--latest` is silently ignored because the `--issue` branch returns early before the unknown-flag check. This is consistent with pre-change behavior and isn't called out by the issue.
5. **Short-branch in usage hint** — `formatMissingTargetError` prints branches as the trimmed shorthand (`investigate-12`) to match the issue's example format. The full branch name (`worktree-dangeresque-investigate-12`) is preserved in the active-worktree field used by other code paths.
