## Risks / Uncertainty

- **Blast radius** — enabling `sandbox_workspace_write.network_access=true` lifts the network-egress denial for *every* spawned shell command under a codex worker, not just web-fetch-shaped ones. A compromised prompt could exfiltrate repo contents via `curl -X POST` or pull arbitrary payloads. Mitigation is documented in the new DESIGN.md paragraph: this is parity with claude's existing `WebFetch` allowlist, the worker is already trusted at the content level via `--full-auto`, destructive shell commands remain blocked via `dangeresque.rules`, and the worktree is isolated. INVESTIGATE §128 called this out explicitly.
- **No live probe** — IMPLEMENT mode did not run `codex exec` itself. Behavior under the new flag is derived from the codex JSON schema + CLI reference + INVESTIGATE artifact, not observed. A VERIFY run should execute e.g. `codex exec --full-auto -c sandbox_workspace_write.network_access=true "curl -sSf https://example.com -o /dev/null && echo OK"` and confirm `OK` (vs failure without the flag).
- **`web_search` default mode is `cached`** — not changed by this PR. Cached results can be stale for fast-moving topics. INVESTIGATE §130 noted `-c web_search=live` as a future-only knob; deferred per the out-of-scope list.
- **`.dangeresque/worker-prompt.md` edit conflicts with "hands off config" rule in AFK_WORKER_RULES.md** — the rule says `.dangeresque/*.md` is human-managed on main, but the staged IMPLEMENT plan explicitly lists this file as a target ("Both files drift in parallel today; the worker reads the deployed copy, the template seeds new projects"). I interpreted the explicit issue authorization as overriding the general rule: the worker reads the deployed `.dangeresque/` copy at runtime per `src/runner.ts` (`configDir = join(projectRoot, CONFIG_DIR)`), so updating only `config-templates/` would leave the new guidance unreachable in the current project until someone re-ran `dangeresque init`. Flagging for human review in case the intent was template-only.
