## Summary

Parity gap is **real but narrower than the issue hypothesized**. Codex is not a blank slate on web research — it ships with a first-class `web_search` tool built into the model runtime that does NOT go through the sandbox. The actual gaps versus claude are:

1. **Shell `curl` / `wget`**: blocked under `--full-auto` because `sandbox_workspace_write.network_access` defaults to `false`. Fixable in one flag: `-c sandbox_workspace_write.network_access=true`.
2. **No named `web_fetch` tool**: codex has `web_search` but no equivalent of claude's `WebFetch(url, prompt)` built-in. Can be bridged by (a) shell curl once network is on, or (b) an HTTP-fetch MCP server (e.g. `modelcontextprotocol/server-fetch`).

Engine routing (Q4) is overkill for this gap. The minimum-viable fix is a one-line addition to `buildCodexWorkerArgs` in `src/runner.ts` plus a prompt clarification so codex workers don't look for a `WebSearch` tool by name.

## Investigation Questions — Answers

### Q1. Does `codex exec --full-auto` enable network by default?

**No.** `--full-auto` is documented as a preset for `--sandbox workspace-write --ask-for-approval on-request` (codex CLI reference). The workspace-write sandbox config `sandbox_workspace_write.network_access` defaults to **`false`** per the JSON schema at `codex-rs/core/config.schema.json` on the codex main branch.

To enable outbound network for shell commands, either:
- **Per-run via CLI flag** (cleanest for dangeresque): `-c sandbox_workspace_write.network_access=true`
- **Project config.toml** (persistent, committed): add `[sandbox_workspace_write]` with `network_access = true` under `.codex/config.toml` at the worktree root
- **User config.toml**: same key in `~/.codex/config.toml` (global — bad fit for dangeresque)

The `-c` flag syntax supports dotted nested keys (confirmed via the codex main-branch CLI source and the official reference). TOML-ish `key=value`, JSON-parsed where possible.

### Q2. Can codex workers reach external URLs via shell `curl`?

**Not under the current runner configuration.** `src/runner.ts:416-434` builds codex args as `["exec", "--json", "--full-auto", "--model", ..., "--cd", worktreePath, prompt]` with no `-c` overrides. Therefore the effective sandbox is `workspace-write` with `network_access=false`, and any spawned shell command making an outbound connection is denied by codex's project-layer exec policy / OS sandbox.

Per the codex sandboxing doc: *"The sandbox applies to spawned commands, not just to Codex's built-in file operations."* That means `curl`, `wget`, `npm install` over network, `gh`, etc. are all blocked when `network_access=false`.

**No live probe was performed** because INVESTIGATE mode does not run workers. A trivial verification for a follow-up IMPLEMENT/VERIFY:

```
codex exec --full-auto -c sandbox_workspace_write.network_access=true \
  "run: curl -sS https://example.com | head -n 5"
```

vs the same without `-c` (expected to fail with a sandbox-denied error).

### Q3. Does codex support MCP and/or a web-fetch MCP?

**Yes, MCP is supported natively.** Configuration is via `~/.codex/config.toml` (global) or `.codex/config.toml` (project-scoped at worktree root), with sections of the form:

```toml
[mcp_servers.<name>]
command = "..."       # stdio transport
args    = [...]
env     = { ... }
# or:
url     = "..."       # streamable HTTP transport
```

There is also a `codex mcp add` CLI subcommand. The README already documents this alignment: *"Codex MCP is configured in `~/.codex/config.toml` under `[mcp_servers]`. Keep entries aligned across both tools if you want equivalent MCP behavior for both engines."* (`README.md:457-461`).

**Built-in tools:**
- `web_search` — a first-class codex tool with modes `"disabled"` | `"cached"` | `"live"`. **Default is `"cached"`** (OpenAI-indexed, does not fetch live pages). `--search` is the CLI shortcut for `-c web_search=live`.
- `tools.web_search` — object form with `allowed_domains`, `context_size` (`low`/`medium`/`high`), and `location` for result refinement.
- **No `tools.web_fetch`** in the JSON schema. This is the narrow gap versus claude's `WebFetch`.

The built-in `web_search` tool runs at the model/agent layer, not as a spawned shell command, so it is **not affected by `sandbox_workspace_write.network_access`**. Codex workers can already do web research today via `web_search` (cached mode) without any sandbox change.

### Q4. Engine routing as fallback?

**Not necessary, and not recommended as the primary fix.** The issue hypothesized that "codex has no web tool and therefore web-heavy issues must route to claude," which overstates the gap:

- Codex `web_search` (built-in) covers the overwhelming majority of "use the web to confirm X" research.
- Shell `curl` covers arbitrary URL fetch once `network_access=true`.
- An HTTP-fetch MCP covers the named-tool `WebFetch(url, prompt)` shape if we want strict symmetry.

Engine routing as a *general* mechanism (per-issue metadata selecting claude vs codex) is a separate design topic worth filing its own issue — useful for other asymmetries (e.g. `--effort`, JSONL log semantics, model-family capabilities) — but it is not the minimum-viable fix for this issue.

## What This Means for Dangeresque

- `src/config.ts:57-71` ships `WebSearch` and `WebFetch` in default `allowedTools`. These are **claude-only tool names** — `bashPatternToPrefixRule` already drops them when building codex rules (see `src/runner.ts:111-114` comment and `test/unit/runner.test.ts:269,317`).

It is the per-issue prompts (e.g. the #38 INVESTIGATE guidance that said "use WebSearch / WebFetch to confirm sandbox flags") that leak claude-isms.

### Recommended fix scope for a follow-up IMPLEMENT

One narrow code change plus two small doc updates.

1. **`src/runner.ts` `buildCodexWorkerArgs` (and mirror in `buildCodexReviewArgs`):** add `-c sandbox_workspace_write.network_access=true` to the arg vector. ## Risks / Uncertainty

- **Blast radius of enabling `sandbox_workspace_write.network_access=true`.** This lifts the network denial for *every* spawned command in the codex worker, not just fetch-shaped ones. A rogue prompt could exfiltrate repo contents via `curl -X POST`, or pull down arbitrary payloads. Mitigations: the `acceptEdits`/`--full-auto` model already trusts the worker at the content level; the worktree is isolated; the adversarial reviewer sees the diff. Claude workers have the same latent risk today since `WebFetch` is in allowedTools. Parity, not amplification — but worth naming.
- **No live probe.** Expected behavior is derived from schema + docs, not a live `codex exec` run. A follow-up VERIFY should actually run `codex exec --full-auto -c sandbox_workspace_write.network_access=true "curl -sSf https://example.com -o /dev/null && echo OK"` and confirm `OK`. I did not run this because INVESTIGATE mode forbids code change and the one-line shim to test it properly belongs in an IMPLEMENT run.
- **`web_search` default mode = `cached`.** Cached uses OpenAI's index. For some tasks (fast-moving docs, obscure blogs, very recent releases), cached may be stale. Worth considering `-c web_search=live` for any long-running research issue, but not a default change.
- **Per-worktree `.codex/config.toml` alternative.** We could also write a `.codex/config.toml` with `[sandbox_workspace_write] network_access = true` alongside the existing `.codex/rules/dangeresque.rules` in `writeCodexRulesFile`. Probably cleaner than `-c` on the CLI because we already generate codex-specific files per worktree — but requires verifying codex's config-layer precedence (does project config override user config? Does codex actually discover `.codex/config.toml` in `--cd`?). `-c` is safer because it's unambiguous. Noting both for the IMPLEMENT decision.
- **No `tools.web_fetch` in codex JSON schema.** Confirmed by reading the schema, but schemas lag features sometimes. If OpenAI adds a native `web_fetch` tool to codex after 0.122.0, the parity gap shrinks further without us doing anything.
- **Sandbox behavior differs by OS.** The codex sandbox doc acknowledges "implementation differs between macOS, Linux, WSL2, and native Windows." The config key `sandbox_workspace_write.network_access` is the portable escape hatch, but actual enforcement (seatbelt vs landlock/seccomp) differs. Not blocking for this issue; a future runtime-matrix test could confirm across OSes.
