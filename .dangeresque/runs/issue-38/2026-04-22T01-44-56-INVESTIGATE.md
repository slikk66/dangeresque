## Summary

Pass 2 investigation of codex sandbox behavior for git commits and yarn tooling from linked worktrees under `--full-auto`.

**Core finding:** codex's sandbox explicitly and intentionally protects `.git` (both the `.git` directory AND the resolved gitdir for linked worktrees) as read-only in every `workspace-write` configuration. This is hardcoded in `codex-rs/protocol/src/permissions.rs::default_read_only_subpaths_for_writable_root` and reaffirmed by OpenAI's public docs. There is **no public config knob** that lets codex commit from inside a linked worktree under `workspace-write`. Only `danger-full-access` or a "rule" that runs `git` outside the sandbox entirely gets around it — both re-inflate the security surface.

**Secondary finding:** yarn tooling already works empirically under `--full-auto`. The codex-implement-28 session log shows `yarn install --immutable`, `yarn build`, and `yarn test` (64 tests passing) all succeeded inside the sandbox. The ONLY failing step was `git add` / `git commit`.

**Recommendation:** **Option A (dangeresque-side commit after codex exits)** is the only viable path for the primary blocker. Yarn does not need any sandbox tweaks for today's use cases. Document network/cache tuning for future cases where new deps are added or cache is cold.

## Investigation Answers

### Q1. Network toggle under workspace-write

**Config key:** `sandbox_workspace_write.network_access` (boolean). Default `false`.

**Command-line override (dot-notation):** `-c sandbox_workspace_write.network_access=true`

**Example (single session):**
```
codex exec --full-auto -c sandbox_workspace_write.network_access=true ...
```

**Does it require `danger-full-access`?** No. `network_access` is an additive toggle inside `workspace-write`. It gates outbound network independently of the file-write sandbox. Confirmed by the [Configuration Reference](https://developers.openai.com/codex/config-reference) and [Sandbox docs](https://developers.openai.com/codex/concepts/sandboxing).

**Cite:** `sandbox_workspace_write.network_access` — type `boolean`, "Allow outbound network access inside the workspace-write sandbox." ([config-reference](https://developers.openai.com/codex/config-reference))

### Q2. Yarn under workspace-write (EMPIRICAL: already works)

**Session-log evidence** from `.dangeresque/sessions/dangeresque-implement-28/worker-codex-2026-04-21T23-31-15-873Z.jsonl`:
- `item_42`: `/bin/zsh -lc 'yarn install --immutable'` → `exit_code: 0` ("Done in 0s 129ms" — everything cached, no network)
- `item_43`: `/bin/zsh -lc 'yarn build'` → `exit_code: 0`
- `item_54`: `/bin/zsh -lc 'yarn test'` → 64 tests pass, `exit_code: 0` per worker's `item_58` message
- `item_59`: "The implementation and tests are verified, but the commit step is blocked by sandbox permissions on the parent repository's git metadata: `git add` cannot create the worktree index lock."

**Why yarn works despite cache being outside cwd:**

1. **Reads are unrestricted** in `workspace-write` — the sandbox only gates writes. Yarn berry's global cache at `~/.yarn/berry/cache/` is readable.
2. **yarn install --immutable** with a warm global cache and a committed `yarn.lock` only needs to: (a) read cache → OK, (b) write `node_modules/` inside cwd → OK, (c) write `.yarn/install-state.gz` inside cwd → OK. No network, no cache writes.
3. **yarn build** runs `tsc` from `node_modules/.bin/` → reads-only outside cwd, writes `dist/` inside cwd → OK.
4. **yarn test** runs `node --test` on compiled dist → all inside cwd → OK.

**writable_roots defaults under workspace-write** (from [sample config](https://developers.openai.com/codex/config-sample) + seatbelt source):
- cwd (always)
- `$TMPDIR` (unless `sandbox_workspace_write.exclude_tmpdir_env_var = true`)
- `/tmp` (unless `sandbox_workspace_write.exclude_slash_tmp = true`)
- Additional roots from `sandbox_workspace_write.writable_roots` or `--add-dir PATH`
- But: **each writable root gets an automatic read-only subpath list of `.git`, `.codex`, `.agents`** — see [permissions.rs](https://raw.githubusercontent.com/openai/codex/main/codex-rs/protocol/src/permissions.rs) `default_read_only_subpaths_for_writable_root`.

**Cache writes only become a problem when:**
- Cold cache (first checkout of new dep) → needs network + cache write.
- Yarn corepack bootstrap if `packageManager` version mismatches host yarn → writes to `~/.cache/corepack/`.

**Mitigations if ever needed:**
1. Flip to project-local cache: add to `.yarnrc.yml`:
   ```yaml
   enableGlobalCache: false
   ```
   Yarn 4 default for `enableGlobalCache` is `true` (→ `~/.yarn/berry/cache/`). Setting to `false` sends cache to `<cwd>/.yarn/cache/` (writable). ([yarn docs](https://yarnpkg.com/configuration/yarnrc))
2. Enable network: `-c sandbox_workspace_write.network_access=true`
3. Both are orthogonal to the commit problem.

### Q3. node_modules in a fresh git worktree

**Confirmed:** `.gitignore` in this repo contains `node_modules/`, so `git worktree add` produces a worktree WITHOUT `node_modules`. The codex implement-28 session confirmed this: `yarn build` failed first ("Yarn has no installed dependency state in this worktree"), then `yarn install --immutable` populated it in 129ms (fully cached), then `yarn build` succeeded.

**Options to handle:**
- **(a) Let the worker run `yarn install --immutable` itself** — current behavior, empirically works when the global cache is warm. Fails if cold cache (would need network).
- **(b) dangeresque-side preinstall** — runs `yarn install --immutable` from the Node process before spawning codex. Costs ~100-500ms extra. Removes one source of worker flakiness. The Node process has full host permissions, so cold-cache / corepack-download scenarios also work.
- **(c) Symlink `<main>/node_modules` → `<worktree>/node_modules`** — saves disk + time. Risk: if main checkout's node_modules doesn't match worktree's lockfile, breaks silently. Not recommended.

**Recommended:** **(a) is sufficient today** because this repo's cache stays warm between runs. If cold-cache failures become common, add **(b)** as a thin pre-step.

### Q4. Final Recommendation (A vs B vs C vs C+network vs hybrid)

**Recommendation: Option A.** Dangeresque-side commit after codex exits.

**Rationale — why C is impossible (not just risky):**

- `codex-rs/protocol/src/permissions.rs` `default_read_only_subpaths_for_writable_root`:
  ```rust
  let top_level_git = writable_root.join(".git");
  // This applies to typical repos (directory .git), worktrees/submodules
  // (file .git with gitdir pointer), and bare repos when the gitdir is the
  // writable root itself.
  let top_level_git_is_file = top_level_git.as_path().is_file();
  let top_level_git_is_dir = top_level_git.as_path().is_dir();
  if top_level_git_is_dir || top_level_git_is_file {
      if top_level_git_is_file
          && is_git_pointer_file(&top_level_git)
          && let Some(gitdir) = resolve_gitdir_from_file(&top_level_git)
      {
          subpaths.push(gitdir);     // <-- EXPLICITLY protects <main>/.git/worktrees/<name>/
      }
      subpaths.push(top_level_git);
  }
  ```
  Codex specifically **detects linked worktrees** (via the `.git` pointer file), **resolves the gitdir** (`<main>/.git/worktrees/<name>/`), and adds THAT to `read_only_subpaths`. This is not accidental — it's designed intent.

- Docs confirm: "`<writable_root>/.git` is protected as read-only whether it appears as a directory or file" with no override ([agent-approvals-security](https://developers.openai.com/codex/agent-approvals-security)).

- Existing GitHub issues confirm this is live and unresolved:
  - [#12280 Allow writable .git/gitdir](https://github.com/openai/codex/issues/12280) — feature request, open, no maintainer response quoted.
  - [#7071 Unable to create .git/index.lock](https://github.com/openai/codex/issues/7071) — same root cause, open Nov 2025.
  - [#6797 codex tries to delete .git/index.lock](https://github.com/openai/codex/issues/6797) — related, open.

- Even if we `--add-dir <main>/.git/worktrees/<name>/` and `--add-dir <main>/.git/objects/` as separate writable roots (theoretically ORs with cwd's deny via seatbelt allow-union), this fights codex's intent, is fragile across upgrades, and only works on macOS (Linux landlock semantics may differ). **Not recommended.**

**Rationale — why not C' (rules file)?**

Codex supports `.codex/rules/default.rules` (Starlark) with `prefix_rule(pattern=["git","commit"], decision="allow")` that runs a command OUTSIDE the sandbox. This would unblock `git commit` BUT:
- Running git outside sandbox gives full host permissions — worker could run `git push`, `git reset --hard`, `git branch -D main`, `git rebase -i`.
- Pattern matching is prefix-based; blocking dangerous sub-invocations requires a long and error-prone deny-list.
- dangeresque-Node already has full host permissions AND already has a small, tight command list (`add -A`, `commit -m`); letting codex invoke git outside sandbox reintroduces a much larger attack surface.
- We'd still need to commit a `.codex/rules/default.rules` file to the repo AND set up `CODEX_HOME` / the rules scan path correctly — an extra moving piece.

**Rationale — why not D (danger-full-access)?**

Defeats the whole point. Codex docs warn: "should only be used inside an externally hardened environment." Nothing in dangeresque's threat model justifies this.

**Option A mechanics (concrete):**

In `src/runner.ts`, the codex branch of `runWorker` has:
```typescript
child.on("close", (code: number | null) => {
  logStream.end();
  removePidFile(worktreePath);
  if ((code ?? 0) === 0) commitArchiveFile(worktreePath, archivePath);
  resolve({ worktreeName, branch, exitCode: code ?? 0, archivePath });
});
```

Add a new step **before** `commitArchiveFile` that captures worker changes:

```typescript
function commitWorkerChanges(worktreePath: string, issueData: IssueData, mode: string): void {
  try {
    // Check if there's anything to commit
    const status = execSync("git status --porcelain", {
      cwd: worktreePath, encoding: "utf-8", stdio: "pipe",
    }).trim();
    if (!status) return;

    execSync(`git add -A`, { cwd: worktreePath, encoding: "utf-8", stdio: "pipe" });
    // -A captures all changes in the worktree; safe because:
    // 1) Worktree is an isolated branch from origin/HEAD
    // 2) .gitignore still applies (node_modules/, dist/, .dangeresque.pid, etc.)
    // 3) Run artifact file isn't staged yet — commitArchiveFile handles that separately

    const msg = `codex ${mode} worker: issue #${issueData.number}`;
    execSync(`git commit -m "${msg}"`, { cwd: worktreePath, encoding: "utf-8", stdio: "pipe" });
  } catch {
    // Either nothing to commit or commit failed — log and continue
    // (commitArchiveFile runs after and handles its own try/catch)
  }
}
```

Call site inside `child.on("close")`:
```typescript
if ((code ?? 0) === 0) {
  commitWorkerChanges(worktreePath, opts.issueData, opts.mode ?? "INVESTIGATE");
  commitArchiveFile(worktreePath, archivePath);
}
```

**Why `-A` is safe scope-wise:**
- Worktree is throwaway until `dangeresque merge`. Every unignored file in the worktree is, by definition, the worker's work product.
- `.gitignore` excludes `node_modules/`, `dist/`, `*.tsbuildinfo`, `.yarn/install-state.gz`, `HANDOFF.md`, `.dangeresque/sessions/`, and `.dangeresque.pid` (via `.dangeresque.pid` and `.gitignore`). Stray build outputs stay out of the commit.
- The worker-prompt.md tells the worker "git add your changes + git commit" — if the worker DID manage to stage-but-not-commit (unlikely in the codex sandbox case), `git add -A` collapses that too.

**Engine parity:** Claude workers already commit themselves (they have host permissions). `commitWorkerChanges` would be a no-op on claude runs when called (nothing new to commit because the worker already did). To keep the code path engine-agnostic but skip the redundant work on claude, guard the call to `commitWorkerChanges` on `opts.config.engine === "codex"`. Simpler and equivalent either way.

**Commit-message quality:** Auto message `codex ${mode} worker: issue #${N}` is terse but honest. For richer messages, future iteration could read the worker's run-artifact summary block and derive a one-liner. Not needed for MVP.

## Observations

1. **Yarn tooling is not the blocker.** Empirical session log proves yarn install/build/test all work under codex `--full-auto`. The original issue's comment #2 ("Yarn under workspace-write (the big one)") worried this was infeasible — it isn't, for this project's common case. This de-risks Option A significantly.

2. **Fresh-worktree node_modules handling already works.** The worker runs `yarn install --immutable` itself when it sees missing deps. Takes ~130ms with warm global cache.

3. **Codex source intentionally protects the linked-worktree gitdir.** Not an accidental sandbox miss — the Rust source has a dedicated branch to resolve `.git` pointer files and add the remote gitdir to read-only paths. Any "fix" via codex configuration would be fighting deliberate design.

4. **Rule-based allowlist is dangerous.** A `.codex/rules/default.rules` with `decision = "allow"` runs commands outside the sandbox entirely. For `git`, this means full host access (pushes, force-resets, refspec deletes, etc.). Dangeresque's existing `disallowedTools` list on the claude side shows the human has already decided these are unsafe for workers. We'd have to re-encode that list as negative-prefix rules, which is brittle.

5. **Claude-engine regression risk is zero.** Option A's new code path is gated on `engine === "codex"`. Claude flow is untouched.
