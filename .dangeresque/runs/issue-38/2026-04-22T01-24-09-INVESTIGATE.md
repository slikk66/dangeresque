# Status

`investigating` — root cause confirmed, two viable remediation paths identified, no code changes.

# Summary

The hypothesis in the issue is **correct**: codex `--full-auto` runs under sandbox `workspace-write`, which **hardcodes `.git` (and the gitdir resolved from a `.git` pointer file) as a read-only sub-path** within any writable root. Linked git worktrees store per-worktree state at `<main-checkout>/.git/worktrees/<name>/`, which is exactly the path codex protects, so `git commit` from within a linked worktree fails with `Operation not permitted: …/.git/worktrees/<name>/index.lock`.

There **is** a sandbox escape (`--dangerously-bypass-approvals-and-sandbox`, alias `--yolo`), but it removes the sandbox in its entirety (filesystem boundary, network restrictions, and approval gating). It is the path the OpenAI maintainer recommends today. The narrower escape that the issue's `writable_roots` hypothesis hopes for **does not exist** — `.git` protection is unconditional within `workspace-write`.

# Verification — investigation evidence

## 1. What `--full-auto` actually is

> "`--full-auto` is the lower-risk local automation preset: `sandbox_mode = "workspace-write"` and `approval_policy = "on-request"`."
> — [OpenAI Codex sandbox docs](https://developers.openai.com/codex/concepts/sandboxing)

So `runner.ts:323-328` (`--full-auto`) is equivalent to `--sandbox workspace-write --ask-for-approval on-request`.

## 2. `.git` is hardcoded read-only under workspace-write

OpenAI sandbox docs:

> "`.git` is protected as read-only whether it appears as a directory or file."
> "If `.git` is a pointer file (`gitdir: …`), the resolved Git directory path is also protected as read-only."

Confirmed in source — `codex-rs/protocol/src/protocol.rs` defines `WritableRoot.read_only_subpaths` containing `.git`, `.codex`, `.agents`. The list is hardcoded; there is no config knob to disable it. Quoted from openai/codex issue #7071 comment by `@gyf1214`:

> "The `subpaths` are read-only sub-directory exceptions even when a directory is writable under `workspace-write`. Right now it includes `.git` (also gitdir when using worktree), `.codex` and `.agents`."

Issue [openai/codex#9313](https://github.com/openai/codex/issues/9313) (CLOSED, fixed) confirms the sandbox now resolves `.git` *file* pointers (used by linked worktrees) and protects the resolved gitdir. So adding `<main-checkout>/.git/worktrees/<name>/` to `sandbox_workspace_write.writable_roots` will not work — the path is still on the protected sub-path list.

## 3. Maintainer-recommended escape

[openai/codex#5034](https://github.com/openai/codex/issues/5034) by OpenAI collaborator `@nornagon-openai`:

> "Eventually we will have more fine-grained control over the sandbox, but for now if you want write access to .git you'll need to pass `--dangerously-bypass-approvals-and-sandbox`."

This flag (alias `--yolo`) is available on `codex exec`. Equivalent `-c` overrides:

```
-c sandbox_mode=danger-full-access -c approval_policy=never
```

Per [CLI reference](https://developers.openai.com/codex/cli/reference), `codex exec` accepts repeatable `-c key=value` overrides where values parse as JSON when possible.

## 4. Other workarounds (rejected as unsuitable for dangeresque)

- **Shadow git-dir** (`GIT_DIR=.aigit codex …` then `git pull .aigit`) — proposed by `@dmahurin` on #5034. Breaks the worktree-as-branch model dangeresque relies on.
- **Tell agent to escalate git** — recommended by maintainer for interactive use, but irrelevant under `--full-auto` (no human present to approve).
- **Pre-create `.git/worktrees/<n>/index` writable somehow** — not supported by codex; the protection is path-based, not permission-based.

# Options & trade-offs

## Option A — switch codex to `--dangerously-bypass-approvals-and-sandbox`

**Change:** In `buildCodexWorkerArgs` and `buildCodexReviewArgs`, replace `--full-auto` with `--dangerously-bypass-approvals-and-sandbox`. Optionally use the explicit pair `-c sandbox_mode=danger-full-access -c approval_policy=never` for clarity in logs.

**Pros**
- Smallest possible diff (one flag per call site).
- Codex commits land naturally as part of the worker run; the worker can iterate (multiple commits, meaningful per-commit messages) just like a claude worker.
- Engine parity with claude — claude already runs without an effective filesystem sandbox under `acceptEdits`, and dangeresque already trusts the worktree as the isolation boundary.
- Mirrors the maintainer-recommended path.

**Cons**
- Removes ALL sandbox protection: filesystem (anywhere on host), network access, approval gating. The worker could in principle exfiltrate via network or touch files outside the worktree. (claude has the same exposure today; codex has been protected, this gives it up.)
- Codex does not honor dangeresque's `allowedTools`/`disallowedTools` config, so the existing claude-side denylist (`git push`, `rm -rf`, `git reset --hard`, `git branch -D`) is **not** mirrored on the codex side. Bypassing the sandbox makes that asymmetry sharper. (Note: even today, with `--full-auto`, codex does not honor those tool denylists — so this is not a regression, just a continuation of an existing gap.)
- The flag name explicitly warns "Only use inside an externally hardened environment." Dangeresque's worktree is *branch* isolation, not *process* isolation.

## Option C — dangeresque-side commit after worker exits (issue's recommended fallback)

**Change:** In `child.on("close")` for the codex worker (`runner.ts:490-496`), before `commitArchiveFile`, run `git -C <worktreePath> add -A` then `git -C <worktreePath> commit -m "<auto-generated>"`. Catch "nothing to commit" gracefully (the worker may have made no changes, e.g. INVESTIGATE).

**Pros**
- Preserves codex's `workspace-write` sandbox during worker execution. The agent stays bounded.
- Surgical: addresses exactly the .git block, nothing else.
- Proven by the manual #28 salvage — the exact same operation, lifted into automation.
- `-A` scope is safe inside the worktree because the worktree is its own branch — there is no "wrong" file to commit.

**Cons**
- One auto-commit per run. Worker cannot produce a multi-commit history with logical separation.
- Commit message must be auto-generated (or read from a worker-written manifest). Worker can't write a meaningful natural-language commit message during its run.
- Adds engine-specific code in `runner.ts` — the claude path doesn't need this. Mild violation of the ONE-PATH directive (`CLAUDE.md`), though scoped to one branch in `runWorker`.
- Edge case: if the worker successfully ran some shell commands that touched the worktree but produced unintended file changes (e.g. cache files, accidental `.DS_Store`), `add -A` sweeps them in. Mitigation: a project `.gitignore` + checking for huge diffs before committing.
- Review pass also needs a parallel commit (review may write/edit files per `runner.ts:256-260` allowedTools), or review commits must continue to use `commitArchiveFile` only. Need to think through review semantics if the review wants to commit notes.

## Option B — granular `-c` overrides (`-c sandbox_mode=danger-full-access -c approval_policy=never`)

Functionally equivalent to A for the .git problem. Slightly more auditable in logs because the keys appear explicitly. Not a separate path, just a stylistic variant of A.

## Hybrid not viable

Mixing A and C doesn't help — A removes the need for C entirely. Either keep the sandbox and add C, or drop the sandbox and skip C.

# Recommendation

**Lean toward Option A** with the explicit `-c` form for auditability:

```ts
return [
  "exec",
  "--json",
  "-c", "sandbox_mode=danger-full-access",
  "-c", "approval_policy=never",
  "--model", opts.config.model,
  "--cd", worktreePath,
  prompt,
];
```

Reasoning:
1. Engine parity with claude — both run effectively unsandboxed inside the worktree boundary.
2. Worker semantics stay natural — codex can iterate, commit logically, write its own commit messages.
3. Smaller surface area of new code in `runner.ts`.
4. Aligns with slikk66's staged-comment preference: "If [a sandbox escape] exists and grants .git-worktree writes, prefer that — keeps the engine responsible for its own commits and avoids divergence from the claude path."

**Open the door to C as a follow-up** if the trust model changes (e.g. dangeresque grows to run untrusted plugin engines, or we want to enforce a narrower codex blast radius separately from claude).

# Risks / Uncertainty

- **Empirical test not run.** I have not actually invoked `codex exec --dangerously-bypass-approvals-and-sandbox` (or the `-c` variant) inside a worktree to confirm `git commit` works. Sandbox-related bash is not in the dangeresque allowlist for this worker, so I could not test directly. The IMPLEMENT/VERIFY phases will need to confirm.
- **Network exposure.** Option A allows codex to make outbound network calls. If model behavior or future plugins ever execute network requests with file-system context, that's a vector. Today there is no evidence codex does this unprompted, but it's now possible.
- **`--ephemeral` and other codex defaults.** I did not audit whether removing the sandbox interacts with the `--json` event stream, `child.stderr` output formatting, or session log capture (`createCodexLogPath`). The IMPLEMENT phase should spot-check that the JSON stream still parses cleanly. I see no reason it wouldn't, but flagging.
- **Review-pass parity.** `buildCodexReviewArgs` (`runner.ts:331-361`) also uses `--full-auto`. The review pass already writes the appended review section into the run artifact and `commitArchiveFile` commits it from the parent process. Whether the review pass needs the same flag swap depends on whether it tries to `git add` / `git commit` from inside codex (e.g. for staging fixes). Check the review-prompt to confirm. If review only appends to the artifact and never commits from within codex, the existing parent-process commit is sufficient and the flag swap is optional for review. Still recommend swapping for consistency.
- **Codex sandbox is OS-specific.** Sandbox enforcement on macOS uses `sandbox_exec`; on Linux uses `landlock`. The `.git` protection is in protocol layer, so OS-agnostic. No further OS-conditional logic expected.
