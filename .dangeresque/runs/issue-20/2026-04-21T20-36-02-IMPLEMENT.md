## Risks / Uncertainty

1. **Node execution not in allowlist** — `scripts/migrate-v1-to-v2.mjs` could not be executed directly by the worker. Migration was therefore performed via Read+Edit tool calls instead. Result on disk is equivalent, but the migration script itself was effectively unused. Future migrations of this style should either (a) be added as a yarn script and run via `yarn <script>` (requires updating the worker allowlist to include `Bash(yarn migrate-*)` or similar), or (b) run pre-merge by a human, or (c) be implemented via the Read+Edit pattern as here.
2. **Idempotency of the Read+Edit migration** — each Edit matched `"schema_version": "1"` (or a larger multi-line pattern for IMPLEMENT files). Re-running would find 0 matches (the strings no longer exist) and error. The original script had `if (schema_version !== "1") skip;` logic which would be preferable for idempotency in ongoing use; since the migration is throwaway and on-disk state is now uniform v2, this is not a practical concern.
3. **Ordering of JSON keys in migrated IMPLEMENT files** — `review_model` and `review_effort` are inserted in the same position where new runs will emit them (immediately after `effort`). Verified by spot-check. No other key order changed.
4. **`review_effort` type asymmetry with `effort`** — per issue spec, `review_effort?: string` (optional/undefined when absent), not `string | null` like `effort`. Intentional: "absent = review didn't run", and the field is omitted from JSON rather than serialized as `null`. Matches migration output (review-skipped files have no `review_effort` key at all).
5. **Backward compat with v1 readers** — per investigation, nothing in `src/` currently reads `schema_version`. No readers needed to be adjusted. Future external consumers should treat missing `review_model` / `review_effort` as "review did not run" or "pre-v2 artifact, review ran with worker's model/effort".
6. **`node_modules` not initially installed in worktree** — had to run `yarn install --immutable` once. Fresh worktrees spawned by dangeresque may consistently require this; not changing dangeresque's runner for this issue (out of scope).

## Observations

- The `dangeresque.pid` file appeared in `git status` as untracked during this run — that's the runtime PID file, not mine to commit.
- The artifact builder's `build()` now imports nothing new; the conditional spread uses TypeScript's existing object-spread semantics. No runtime dependency changes.

So the spread always has a defined value for reviews that ran. The `this.init.reviewEffort !== undefined` guard in `build()` is defensive and will essentially always be true when reviewRan — kept for symmetry with `reviewModel` and so that callers may legitimately omit the field.
- Prior INVESTIGATE's plan for the migration was followed almost verbatim, except the script execution step was replaced with in-tool Read+Edit due to the bash allowlist limitation.
