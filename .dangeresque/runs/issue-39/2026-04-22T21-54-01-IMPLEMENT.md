## Status

## Summary

Closed the engine-parity gap #39 identified by translating
`config.disallowedTools` into codex-native command denials.

Each `Bash(<cmd> *)` pattern is translated into a Starlark
`prefix_rule(pattern=[tokens], decision="forbidden", justification="...")`
line and written to `<worktree>/.codex/rules/dangeresque.rules` once,
pre-spawn, in the codex branch of `runWorker`. Codex's project-layer
rules scan (`codex-rs/core/src/exec_policy.rs::load_exec_policy`) picks
the file up and refuses the matching argv at exec time — giving codex
the same hard-block posture the claude engine gets via
`--disallowed-tools`.

Prompt strings in `buildCodexWorkerArgs` / `buildCodexReviewArgs` were
left untouched per the IMPLEMENT directive — the rules file is the
enforcement; any prompt-level "safety contract" would be redundant.

The generated rules file is excluded from `commitWorkerChanges` so it
does not leak into the merge diff at `dangeresque merge` time (it's a
derived artifact from `.dangeresque/config.json`, regenerated per run).

## What Changed

### `src/runner.ts` — new exports + wire-in

- `bashPatternToPrefixRule(pattern: string): string | null` — translator.
  Matches `^Bash\((.*)\)$`, strips trailing ` *` glob (claude's literal-prefix
  marker), tokenizes by whitespace, emits
  `prefix_rule(pattern=["tok1", "tok2", ...], decision="forbidden", justification="...")`.
  Non-`Bash()` patterns return `null`.
- `buildCodexRulesContent(disallowedTools: string[]): string | null` —
  renders the full rules-file text from a list of patterns, with a
  provenance header comment. Returns `null` when nothing translates.
- `writeCodexRulesFile(worktreePath, disallowedTools): string | null` —
  writes to `<worktree>/.codex/rules/dangeresque.rules`, creating
  intermediate directories. Returns the absolute path written, or `null`.
- `CODEX_RULES_RELPATH = ".codex/rules/dangeresque.rules"` — exported
  constant, reused by `commitWorkerChanges` pathspec and tests.
- `runWorker` codex branch (line 557): calls `writeCodexRulesFile(...)`
  immediately after `createWorktree` resolves and before
  `buildCodexWorkerArgs`, so the file is on disk before the codex process
  starts. One call covers both worker AND review (same worktree, same
  file) — `runReview` is intentionally not modified.
- `commitWorkerChanges` pathspec extended to exclude
  `.codex/rules/dangeresque.rules` alongside the existing
  `.dangeresque/runs/` exclusion.
- `writeFileSync` added to the `node:fs` import line.

2. **Constant placement.** `CODEX_RULES_RELPATH` is declared after
   `commitWorkerChanges` but referenced in its body. Safe because the
   reference is inside the function body — by the time
   `commitWorkerChanges` runs, all module-level `const` declarations
   have been initialized. If a future refactor moves things around, this
   still holds.

3. **Rules-file placement vs. git tracking.** The file is written to an
   untracked path and actively excluded from `commitWorkerChanges`.
   Rationale: the file is derived from config and regenerated per run;
   committing it would pollute the merge diff and create duplicate
   sources of truth. Codex reads it off disk regardless of git status.

4. **Review-phase coverage.** `runReview` does not write the rules file.
   Intentional — same worktree as the worker, same `.codex/rules/`
   folder, so codex picks it up for the review session too. Rebase
   between worker and review doesn't delete untracked files.

5. **Pattern-matching parity caveat (documented in the investigation,
   not re-documented here).** `prefix_rule(pattern=["rm", "-rf"])` is
   token-positional — it does not catch `rm -fr X`, `rm -r -f X`, or
   `rm --recursive --force X`. Claude's `Bash(rm -rf *)` glob has the
   same literal-prefix limitation. Parity at the *pattern level* is
   what's achieved; parity at the "human intuition" level is out of
   scope and the same under either engine.

## Risks / Uncertainty

1. **Sensitive-file block is a partial-delivery risk.** The `.claude/CLAUDE.md`
   line was in-scope per the issue directive but not reachable under AFK
   policy. Human needs to apply the one-line replacement shown in
   Observation #1. Without it, that one doc line still says "hard-blocked"
   without engine qualifier — not *false* (it now is hard-blocked under
   both engines), but less precise than the other three sites.

2. **`decision="forbidden"` version compatibility not gated.** Per
   directive, no `codex --version` check was added. If an old codex
   binary silently drops the rule, the user gets no enforcement and no
   warning. The directive notes this is a separate issue to file if
   observed.

3. **Project-layer scan path not empirically verified.** The write lands
   at `<worktree>/.codex/rules/dangeresque.rules` based on OpenAI's
   documented project-override pattern (`.codex/config.toml` → by symmetry
   `.codex/rules/*.rules`). If codex's scan looks somewhere else, the
   file sits unused. No test in this session exercises codex itself.

4. **Non-Bash entries silently dropped.** `config.disallowedTools` may
   contain entries like `WebSearch`, `Edit`, `mcp__foo` — these have no
   meaning under codex's shell-command rules. The translator returns
   `null` for them and `buildCodexRulesContent` filters them out.
   Documented implicitly by the `bashPatternToPrefixRule: non-Bash()
   patterns return null` test. No startup warning when all entries get
   filtered to null (would produce `null` content → no file written →
   silently no enforcement). Acceptable because non-Bash denylist
   entries are a config-shape choice, not a safety default.

5. **File-write error paths.** `mkdirSync(..., { recursive: true })` and
   `writeFileSync` will throw on permission errors. `writeCodexRulesFile`
   lets the error propagate — in `runWorker` this aborts the run before
   spawning codex, which is safer than proceeding without the rules file
   (a quiet no-op would restore the original engine-parity gap). No
   try/catch added per NO-BANDAID.
