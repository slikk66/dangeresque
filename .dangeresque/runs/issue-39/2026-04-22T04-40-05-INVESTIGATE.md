## Summary

**The bug is real.** `config.disallowedTools` is silently dropped on the codex path. `buildCodexWorkerArgs` (now at `src/runner.ts:357-375` after #38 shifted line numbers; previously 311-329) and `buildCodexReviewArgs` (now at `src/runner.ts:377-412`) build the codex command line with no denylist whatsoever — only `exec --json --full-auto --model ... --cd ... <prompt>`. The claude path wires `config.disallowedTools` through `--disallowed-tools` on both worker (`src/runner.ts:265-267`) and review (`src/runner.ts:306`) args. Engine parity is broken.

**Codex has no CLI flag for command-denylist.** Verified against OpenAI's public docs ([cli/reference](https://developers.openai.com/codex/cli/reference)) and the codex-rs source — the complete `codex exec` flag set is `--cd, --color, --dangerously-bypass-approvals-and-sandbox, --ephemeral, --full-auto, --image, --json, --model, --oss, --output-last-message, --output-schema, --profile, --sandbox, --skip-git-repo-check, -c/--config`. None of these accept a deny-list.

**Codex DOES have a command-denial mechanism, but it is file-based.** `prefix_rule(pattern=[...], decision="forbidden")` rules (Starlark) in a `.rules` file under a scanned config layer deny the command before execution. Confirmed from `codex-rs/core/src/exec_policy.rs::load_exec_policy`: rules are loaded from every config-stack layer's `config_folder().join("rules")` directory, and a forbidden-decision match routes through `ExecApprovalRequirement::Forbidden` (see `derive_forbidden_reason` at line 919-939). Project-local layers are explicitly scanned unless `config_stack.ignore_user_and_project_exec_policy_rules()` is set.

**Sandbox modes (`read-only`, `workspace-write`, `danger-full-access`) are filesystem+network policies, NOT command-string matchers.** From the docs: "sandbox behaviors are 'filesystem writes + network' related rather than command-string inspection." So `rm -rf` inside cwd is allowed under `workspace-write`, `git reset --hard` on local refs is allowed (it writes to `.git/` which is the main-checkout gitdir — and for linked worktrees, that gitdir IS read-only per #38's pass-2 finding, so `reset --hard` actually IS partially blocked by filesystem policy — but not reliably and not by intent), and `git push` is incidentally blocked by the network policy rather than by any command filter.

**The codex worker prompt currently gives zero guidance.** `grep` through `.dangeresque/worker-prompt.md` and the `.dangeresque/review-prompt.md` for `rm -rf|git push|git reset|git branch -D|disallowed` yields zero matches. Option (a) from the issue is not implemented — not "weak-but-present," actually absent.

**Docs overstate the safety guarantee.** `README.md:437`, `docs/DESIGN.md:62-63` ("hard-blocked at the Claude Code tool layer, regardless of what the prompt says"), `docs/DESIGN.md:103-104`, and `.claude/CLAUDE.md:62` all claim these commands are hard-blocked for "workers" without distinguishing engine. Under `engine: codex`, this claim is currently false.

## Investigation Answers

### Q1 — Does `codex exec` support a command-denylist flag?

**No CLI flag.** `codex exec --help` was not directly runnable here (bash permissions denied `codex` as a command), but two independent sources confirm:

1. **OpenAI's CLI reference** ([cli/reference](https://developers.openai.com/codex/cli/reference)) enumerates every `exec` flag. No command-denylist flag exists. The closest is `-c/--config` for inline TOML dot-notation overrides — but rules are NOT configurable inline (see Q3).
2. **Codex-rs source** (`codex-rs/core/src/exec_policy.rs::load_exec_policy`) loads rules from files only, iterating config-stack layers and their `rules/` subdirectories.

### Q2 — If not, which of the four options is viable?

**Option (a) — prompt-level refusal:** NOT CURRENTLY IMPLEMENTED. The codex worker prompt (`config.workerPrompt`) is loaded at `src/runner.ts:359-360` and concatenated with task-prompt text — neither side mentions these banned commands. Adding a paragraph is easy but unenforceable. Codex is a persuasive-text-following agent; under adversarial prompt injection or just plain confusion, it will run a banned command if the task seems to need it. Strength: near-zero defensive value. Cost: trivial.

**Option (b) — PTY wrapper:** Technically possible but a bad fit. Codex streams structured JSON events on stdout (we already parse them in `src/logs.ts`) — wrapping stdout in a pattern-matching killer process is racy (command string appears in log AFTER execution starts or even completes), requires a second process per run, and has no clean hook for "decide before execute." Do not recommend.

**Option (c) — narrow the codex sandbox:** IMPOSSIBLE via sandbox alone. Codex sandbox modes are filesystem+network policies, not command-string matchers. Dropping to `--sandbox read-only` would block ALL writes — IMPLEMENT mode stops working entirely. There is no "workspace-write minus some shell commands" intermediate. The claim in the issue body that this might be "what #38 produces" does not match the situation: #38 diagnosed the sandbox as TOO tight for .git writes; fixing that loosened things, not tightened them. Sandbox cannot solve this.

**Option (d) — document + accept:** The minimal honest fix. The current state is that docs claim hard-block safety that doesn't exist under codex. Fixing the docs — plus optionally a startup warning — restores truth without changing behavior.

**Option (e) — NEW: inject a per-run codex rules file (discovered during investigation).** This is the option the issue did not list. See "Recommended Path" below.

### Q3 — What does `codex exec --help` expose?

I could not run the command directly (bash permission denied; codex is not in the `allowedTools` list, and it is not in the default config either). Relying on two indirect sources:

1. **Public docs** — cli/reference page — list every flag. Complete enumeration above.
2. **Prior pass on #38** — confirmed `-c key=value` dot-notation overrides work and cited `-c sandbox_workspace_write.network_access=true` as a working example. But `-c rules.prefix_rules=...` is explicitly NOT supported (from config-reference: rules live in `requirements.toml` or `.rules` files, not inline overrides).

**Recommendation for future:** add `Bash(codex --help)` and `Bash(codex exec --help)` to the worker's `allowedTools` so future investigators can pin exact current-version output. Out of scope for this issue.

### Phase 2 — Real parity via codex rules file (follow-up issue, not this one)

Translate `config.disallowedTools` → a `prefix_rule(..., decision="forbidden")` block, write it to `<worktree>/.codex/rules/dangeresque.rules` during worktree creation (before spawning codex), let codex pick it up via project-layer scan.

**Translator sketch** (Bash-pattern → prefix_rule):
- Parse `Bash(git push *)` → tokens `["git", "push"]` → `prefix_rule(pattern=["git","push"], decision="forbidden", justification="dangeresque: git push blocked (no remote writes from workers)")`.
- Parse `Bash(rm -rf *)` → tokens `["rm", "-rf"]` → `prefix_rule(pattern=["rm","-rf"], ...)`.
- Parse `Bash(git reset --hard *)` → tokens `["git", "reset", "--hard"]`.
- Parse `Bash(git branch -D *)` → tokens `["git", "branch", "-D"]`.

**Pattern-semantics caveats (to document):**
- Codex prefix-rule matching is token-positional. `rm -rf X` matches `["rm","-rf"]` but `rm -fr X` or `rm -r -f X` do NOT. Claude's `Bash(rm -rf *)` glob has the same literal-prefix limitation, so this is parity not regression. Document in the rules file as `# note: `rm -fr` / `rm --recursive --force` not covered; add explicit rules if needed`.
- Rules can accept `any_of` alternatives for token unions, e.g. `prefix_rule(pattern=["rm", any_of("-rf","-fr","-r","--recursive")], decision="forbidden")` — future iteration could widen.

**Deployment caveats (must verify in IMPLEMENT):**
- The exact filesystem path the project-layer scans. From `exec_policy.rs`, rules live under `<layer>.config_folder() + /rules/`. The project layer's `config_folder()` is NOT confirmed from the docs — best guess is `<cwd>/.codex/` based on the `.codex/config.toml` pattern documented for project overrides, but needs empirical check. A one-command live test during IMPLEMENT: write a rules file with a forbidden rule for `touch .proof`, run a codex session that tries `touch .proof`, see if it's refused.
- The minimum codex version that implements `decision="forbidden"`. Public docs confirm the value exists; whether every in-the-wild codex binary supports it is unknown. Phase 2 IMPLEMENT should `codex --version` check and fall back to Phase 1 warnings if the rules aren't applied.
- Cleanup: drop the rules file from the worktree's gitignore so it doesn't pollute commits, OR place it outside the worktree by setting `CODEX_HOME=<tmpdir>` in the spawn env. CODEX_HOME-override approach is more isolated but complicates authentication (codex stores OAuth tokens in `~/.codex/auth.json` — redirecting CODEX_HOME would strand auth). Recommend project-local `.codex/rules/dangeresque.rules` + `.gitignore` entry.

**Alternative Phase-2 path — Docker sandbox with container-level command interception.** Explicitly out of scope per DESIGN.md's host-native stance. Not a fit.

## Risks / Uncertainty

1. **Option (e) rules-file approach depends on project-layer scan path.** Strong circumstantial evidence (`exec_policy.rs::load_exec_policy` scans every layer's `rules/` dir; project-layer `.codex/config.toml` is documented) but NOT directly confirmed for `.codex/rules/`. A Phase 2 IMPLEMENT must verify empirically before committing. If project-layer scan doesn't exist or scans a different path, fallback is `CODEX_HOME=<worktree>/.codex-home/` with auth-token copying — more fragile.

2. **`decision="forbidden"` may not be supported by older codex versions.** Docs confirm it for the current version but no explicit "since version X" annotation. Phase 2 should `codex --version` gate and warn if below the known-good floor.

3. **Prompt-level admonition (Phase 1 part 2) is near-useless on its own.** Adversarial prompt injection via issue body or comments can override it. If we ship Phase 1 alone and nobody picks up Phase 2, the actual safety surface is: worktree isolation + human merge gate. That is a legitimate position — DESIGN.md already lists this as Layer 1 and "The human is the merge gate" — but we must be honest that the tool-layer block (Layer 2) is CURRENTLY claude-only. Docs updated in Phase 1 reflect this truthfully. Shipping Phase 1 without Phase 2 is defensible. Shipping NOTHING leaves the README lying.

4. **The `Bash(<cmd> *)` → prefix_rule translator has non-trivial edge cases** (future Phase 2 concern): commands composed via `sh -c '...'`, aliases, `&&` chains, here-docs — codex's prefix-rule matcher sees the literal argv it is about to exec, so these would bypass the rule unless the shell itself breaks them into separate invocations. Same caveat as claude's `Bash()` pattern; not a regression. Document.

5. **Parity is partial.** Even with Phase 2, token-prefix rules don't catch `rm -fr X` when only `rm -rf` is listed. Claude's pattern matcher has the same limitation (glob on literal prefix). So parity at the PATTERN level is achievable; parity at the "human intuition of what should be blocked" level is not. This is a codex Q (which the issue already flagged — "worktree isolation + human review as the safety boundary" in option d). Phase 1 docs should reaffirm worktree + merge gate as the LAST line of defense under either engine.

6. **Review-phase codex.** `buildCodexReviewArgs` also lacks the denylist. Review workers shouldn't run destructive commands because they only append to the artifact file, but nothing currently enforces that. Phase 2's rules file applies to both worker and review spawns (same worktree); no extra work needed. Phase 1 warning should mention review too.

7. **`--dangerously-bypass-approvals-and-sandbox` flag exists.** If any dangeresque config path ever sets this (currently none does), the entire discussion moot. Grep shows no use in the repo today. Phase 1 could add a guard in `loadConfig` that refuses to accept that flag via config override, but this is belt-and-suspenders and arguably YAGNI.
