## Proposed Design (for IMPLEMENT)

### Data model

Extend `StatsSummary` (`src/stats.ts:43-53`) additively — keep `workerDurationsMs` and `totalDurationsMs` unchanged for backwards compat (the unit test `"computeStats: median + p95 durations"` asserts exact values on them):

```ts
export interface StatsSummary {
  total: number;
  byResult: Record<ResultClassification, number>;
  byVerdict: Record<ReviewerVerdict, number>;
  byEngine: Record<string, number>;
  byMode: Record<string, { total: number; success: number }>;
  byModel: Record<string, number>;
  failureCategories: Record<string, number>;
  workerDurationsMs: { median: number; p95: number };      // unchanged — overall
  totalDurationsMs: { median: number; p95: number };       // unchanged — overall
  byModeDurations: Record<string, {                        // NEW
    worker: { median: number; p95: number };
    total: { median: number; p95: number };
    reviewRanCount: number;  // artifacts in this mode with review actually executed
  }>;
}
```

Artifact schema version does NOT change — this is an additive field on the derived summary, not on `RunArtifact`.

### Aggregation (in `computeStats`)

While iterating artifacts, additionally accumulate three per-mode maps:

```ts
const workerByMode: Record<string, number[]> = {};
const totalByMode: Record<string, number[]> = {};
const reviewRanByMode: Record<string, number> = {};

for (const a of artifacts) {
  // ...existing aggregation...
  if (a.worker && typeof a.worker.duration_ms === "number") {
    (workerByMode[a.mode] ??= []).push(a.worker.duration_ms);
  }
  if (typeof a.duration_ms === "number") {
    (totalByMode[a.mode] ??= []).push(a.duration_ms);
  }
  if (a.review !== null && a.review && !a.review.skipped) {
    reviewRanByMode[a.mode] = (reviewRanByMode[a.mode] ?? 0) + 1;
  }
}
```

Then build `byModeDurations` by iterating `Object.keys(byMode)` and computing median+p95 from the three maps (default to 0 / count 0 for missing modes). This guarantees `byModeDurations` has an entry for every mode that appears in `byMode`, even if all its duration fields are absent.

### Render (in `formatStats`)

Replace the current `Durations (ms):` block (`src/stats.ts:280-288`) with:

```
Durations (ms):
                              median           p95
  overall worker             235,050       790,053
  overall total (run)        115,464     1,024,927

  INVESTIGATE worker           8,250        14,500
  IMPLEMENT worker           490,000       790,053
  IMPLEMENT total (run)      670,000     1,082,309
```

Rules:
- Rename the existing two overall rows from `worker` / `total (run)` to `overall worker` / `overall total (run)` (prefix disambiguates from per-mode rows).
- Blank separator line between overall and per-mode sub-block.
- Per-mode ordering: match the existing `By mode (success rate)` sort — `byMode[m].total` desc, tiebreak `localeCompare` asc (`src/stats.ts:240-242`). Consistency with other sections lets a reader find the same mode in the same position.
- For each mode:
  - Always render `<MODE> worker`.
  - Render `<MODE> total (run)` only when `byModeDurations[mode].reviewRanCount > 0`.
- Skip the entire per-mode sub-block (and the blank separator) when `byModeDurations` is empty (zero artifacts or all artifacts lack timing data).

## Risks / Uncertainty

- **Row label rename risk**: changing `worker` → `overall worker` in the printed output is technically a user-facing change. No test asserts the old label exactly, and the issue's example explicitly uses the `overall` prefix, so this is intended. Noted as a behavior delta for a reviewer.
- **Ordering question**: the issue example shows `INVESTIGATE` before `IMPLEMENT`, which is neither alphabetical nor "by count". I recommend matching the existing `By mode (success rate)` section's sort (`total` desc, then alpha). The issue's example is illustrative, not normative, and consistency with sibling sections is more valuable than matching the example literally. IMPLEMENT worker may choose otherwise; document the choice.
- **Custom mode names (80-col)**: projects may define custom modes beyond the five standard ones. A 30-char mode name + ` total (run)` = 42-char label would still fit at 80 cols with 2+42+12+14=70, but a user with a 50-char mode name could overflow. Not a real concern — existing test `"line over 80 cols"` assertion will catch any regression if a future project stretches it.
- **`median`/`percentile` on length-1 array**: already handled by existing helpers (`src/stats.ts:321-334`) — returns the single value for both.
- **`byMode` field already exists with a different shape** — `Record<string, { total, success }>` for the "By mode (success rate)" section. The new field must be `byModeDurations`, not `byMode.durations` or similar, to keep the two sections independent.
