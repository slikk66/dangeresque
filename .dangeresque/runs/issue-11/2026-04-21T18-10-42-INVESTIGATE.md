# Investigation: Codex worker runs don't receive worker-prompt.md or AFK_WORKER_RULES.md

## Status

`investigating` — root cause confirmed, hypothesis matches code.

## Summary

Issue #11's hypothesis is correct. The Claude engine wires `worker-prompt.md` and `review-prompt.md` into the model's system prompt via `--append-system-prompt-file`. The Codex engine skips both steps entirely — nothing in `buildCodexWorkerArgs` or `buildCodexReviewArgs` references `config.workerPrompt`, `config.reviewPrompt`, or `.dangeresque/AFK_WORKER_RULES.md`.

Additionally, the shared task prompt at `src/runner.ts:131` and `src/runner.ts:144` tells the worker to "Follow .dangeresque/AFK_WORKER_RULES.md (**appended to your system prompt**)". That claim is false for Codex — the file is not appended anywhere. A strict-reading Codex worker is told the rules are in its context when they are not.

### What Codex actually loads

Codex's only auto-loaded context is `AGENTS.md` (verified present at the repo root):

Read that file first and apply every rule in it.

After reading, include `[[PROJECT-RULES-LOADED]]` somewhere in your run result file so we can verify compliance.

`AGENTS.md` redirects to `.claude/CLAUDE.md`, which covers project-level rules (build, conventions, architecture) but does NOT contain:
- The `<!-- SUMMARY -->` block template (only in `worker-prompt.md`)
- The Startup Sequence steps (only in `worker-prompt.md`)

- Allowed status language, directive overrides, file-scope enforcement (only in `AFK_WORKER_RULES.md`)
- The `[[PROJECT-RULES-LOADED]]` instruction — technically this IS in `AGENTS.md`, so Codex would still emit it if it followed AGENTS.md. That's the only piece that survives.

For Codex: false — nothing is appended. A worker that strictly trusts "already appended" would never re-read the file.

## Impact

The issue's stated impact is accurate:
- Codex workers do not see the `<!-- SUMMARY -->` block template → artifact verdict parser (`src/artifact.ts`) may not find the block.
- Codex workers do not see the allowed status language table → may emit "fixed"/"done" despite the rule.
- Codex workers do not see mode-specific behavior → INVESTIGATE mode may write code, VERIFY mode may add features.
- Codex reviewers do not see the adversarial-review checklist, the Claims Check procedure, or the output format → review findings may not append to the correct file.

This breaks `dangeresque results`, the reviewer's claims-check, and the JSON artifact's verdict parser.

## Proposed Fix (for IMPLEMENT follow-up)

Match issue scope:

1. **`buildCodexWorkerArgs`** — read `config.workerPrompt` and `.dangeresque/AFK_WORKER_RULES.md`; prepend their contents to the task prompt (Codex has no `--append-system-prompt-file` equivalent, so inline at the top of the user prompt).

2. **`buildCodexReviewArgs`** — read `config.reviewPrompt`; prepend to the review prompt.

3. Consider: update `buildTaskPrompt` so the "appended to your system prompt" phrasing matches reality across engines (e.g. "included at the top of this prompt"), OR leave the phrasing as-is after (1)(2) restore the appended-content invariant. Slight preference for leaving phrasing unchanged once content is restored — keeps scope tight.

File read paths already exist:
- `config.workerPrompt` + `join(projectRoot, CONFIG_DIR, config.workerPrompt)` — same pattern as `src/runner.ts:168`.
- `config.reviewPrompt` + `join(projectRoot, CONFIG_DIR, config.reviewPrompt)` — same pattern as `src/runner.ts:216`.
- `AFK_WORKER_RULES.md` is at `join(projectRoot, CONFIG_DIR, "AFK_WORKER_RULES.md")`. It has a fixed filename (no config entry — confirmed by grepping `config.ts` — there's `workerPrompt` and `reviewPrompt` but no rulesFile field).

## Risks / Uncertainty

- **Prompt size**: Inlining `worker-prompt.md` (~112 lines) + `AFK_WORKER_RULES.md` (~80 lines) into every Codex user prompt adds ~5-10KB of tokens per run. Likely acceptable; note for IMPLEMENT.
- **Double-instructions**: Inlining into the user prompt means the worker may also see `AGENTS.md` → `CLAUDE.md` during its startup reads. Not harmful, but could confuse "system prompt" references in the inlined content. Wording inside `worker-prompt.md` already says "Your initial prompt specifies ..." rather than "Your system prompt specifies ...", so most of it is neutral.
- **Hands-off config rule**: `.dangeresque/*.md` and `config.json` are in the "do not modify without an issue" list. This investigation does not touch them. IMPLEMENT phase would only modify `src/runner.ts` — no config or prompt edits.
- **`reviewPrompt` is already built from `buildReviewPrompt`**: For Claude, the user-prompt is `buildReviewPrompt(...)` AND `review-prompt.md` is appended to system. For Codex, prepending `review-prompt.md` content to `buildReviewPrompt(...)` preserves that separation in spirit.
