### 5. Formula choice — behavioral decision point

Issue proposes: `review_coverage_percent = (1 - skipped/total) * 100`.

Current formula: `(accept + reject + needs_human_review) / total`.

These differ when `unknown > 0`:

- Total verdict population: `accept + reject + needs_human_review + skipped + unknown = total`
- Issue formula counts as "reviewed": `accept + reject + needs_human_review + unknown`
- Current formula counts as "reviewed": `accept + reject + needs_human_review`

`unknown` per `src/artifact.ts:267-282` covers three sub-cases:
1. Worker exited non-zero → review never ran (reviewer DID NOT run)
2. Worker succeeded but archive missing → review may or may not have run
3. Review ran, produced output, but `**Verdict:**` regex didn't match (reviewer DID run)

Issue formula lumps all three into "reviewed." Current formula excludes all three. Neither is a perfect semantic fit, but issue formula matches the example output (`12 + 13 = 25` = total) and is simpler to explain.

**Recommendation for IMPLEMENT:** Adopt the issue's formula `(1 - skipped/total) * 100`. Document in a one-line comment that `unknown` is treated as "reviewed" for this derived metric because auto-skipping is the dominant reason a run lacks a verdict, and lumping in rare parse-failures keeps the invariant `reviewed + skipped = total` (which makes the "12/25 runs reviewed, 13 auto-skipped" output arithmetically self-consistent).

### 6. Proposed output format wording

Issue example:
```
Reviewer coverage:    48% (12/25 runs reviewed, 13 auto-skipped)
```

Caveat: calling the skipped bucket "auto-skipped" is slightly misleading because `--no-review` runs also land there. Options:
- **A.** Use "skipped" (neutral; covers both): `Review coverage: 48.0% (12/25 runs reviewed, 13 skipped)`.
- **B.** Use "auto-skipped" as-written in the issue (dominant case but slightly lossy for `--no-review`).
- **C.** Break it into two counters: `13 skipped (12 auto, 1 --no-review)` — uses `review.skip_reason`, most precise but widens scope.

**Recommendation:** Option A. It's the minimal-surprise phrasing, avoids the misleading "auto" qualifier when a run was manually skipped, and keeps the change tight. Still matches the issue's numeric intent.
