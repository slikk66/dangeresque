## Summary

Confirmed the onboarding gap. `dangeresque --help` (src/cli.ts:40–90) enumerates commands with one-line descriptions; `dangeresque init` "Next steps" (src/init.ts:169–176) hands the user a 4-step post-install checklist. The canonical prose lives in `README.md` (495 lines), but a fresh LLM dropped into a dangeresque-using project has no CLI-reachable path to it — the README only ships in the dangeresque package itself, not in the consumer project.

Recommendation: ship `dangeresque brief`, a single subcommand that prints a self-contained workflow primer (~130 lines of markdown) to stdout. Content versions with the binary. `--help` and `init` "Next steps" both reference it. No flags in v1.

## Findings

### 1. Command naming

Recommend: **`brief`**.

Alternatives considered:
- `brief` — crisp, 5 chars, the issue's own framing. Reads well in `dangeresque brief | less` / `dangeresque brief >> CLAUDE.md`. Winner.
- `guide` — clearer to a reader unfamiliar with the term; close runner-up. Switch if maintainer prefers "no jargon."
- `for-llm` / `ai-brief` — explicit target but verbose; "ai" is overloaded. Reject.
- `help --long` — tempting but `--help` is already overloaded with per-command usage. Keeps concerns separate if kept distinct.

### 4. Flag shape

Recommend v1 has **no flags** beyond the implicit `--help`.

- `--markdown` / `--plain` — no use case yet. Output is always markdown; humans can pipe through `| fmt -s` or a markdown renderer. Don't build toggles speculatively.
- `--section <name>` — tempting for "just the workflow loop" or "just the mode table," but speculative. Ship the monolith; add sectioning only when there's demonstrated demand.
- `--version` / version line — not a flag; **bake a version stamp into the output itself** (read `package.json` `version` via `createRequire(import.meta.url)("../package.json").version`). This makes stale CLAUDE.md pastes self-diagnosing.

### 5. Init integration

Recommend: **opt-in pointer, not auto-append.**

`initProject` (src/init.ts:143–146) already detects whether the project has a `CLAUDE.md`. Two behaviors to wire up:
- When `CLAUDE.md` is absent: existing log already suggests `.dangeresque/CLAUDE.md.sample`. Add one line: "then run `dangeresque brief >> CLAUDE.md` to embed the workflow primer."
- When `CLAUDE.md` exists: add the same tip to the "Next steps" block (src/init.ts:169). Don't modify the user's file.

Why not auto-append:
- CLAUDE.md is user-owned; silent mutation is surprising and undoes the user's own formatting/ordering.
- Re-running `init` should be idempotent; dedupe logic for "did I already append this brief?" adds complexity and drifts as the brief evolves.
- A visible suggestion that the user can run once is easier to reason about and doesn't require tracking append state.

Concrete wording for `init.ts` "Next steps" — insert as a sub-bullet under step 1:

```
  1. Customize CLAUDE.md with your project's rules (workers read this first)
     Tip: `dangeresque brief >> CLAUDE.md` — adds a self-contained workflow primer
```

Also add to `usageForEngine` (src/cli.ts:54–67) `Commands:` block:

```
  brief                                Print a self-contained workflow primer (pipe to CLAUDE.md or less)
```

## Minimum Viable Content — draft outline of the brief text

For the IMPLEMENT to work from, here is the skeleton the brief should follow. This is NOT final copy; it's a structural plan:

```markdown
# Dangeresque Brief

Dangeresque runs AI coding agents (Claude Code or Codex) AFK in isolated git
worktrees. This brief is the self-contained workflow primer — an LLM or human
reading only this document can drive dangeresque correctly end-to-end.

## The Loop

```
INVESTIGATE → read → discuss → stage → merge → push →
IMPLEMENT   → read → discuss → merge → push → (VERIFY)
```

For a well-scoped issue, skip straight to IMPLEMENT:
```
IMPLEMENT → read → merge → push
```

## The One Hard Rule

**Push `main` to origin AFTER every merge, BEFORE dispatching the next run.**
Worktrees branch from `origin/main`. If local main is ahead, the next worker
starts stale and the reviewer flags phantom regressions.

## Creating Issues

[template + example]

## Dispatching a Run

[`dangeresque run --issue N --mode MODE` + defaults]

## Reading Results

[`dangeresque results <short-branch>` / `--issue N` / `--all`]

## Staging Guidance

[`dangeresque stage N --comment "…" --mode IMPLEMENT`]

## Merge or Discard

[`dangeresque merge <short-branch>` / `dangeresque discard <short-branch>`]

## Monitoring

[`dangeresque logs <short-branch> [-f] [--review]`]

## Modes (one-liner each; full semantics in AFK_WORKER_RULES.md)

[5-row table]

## What NOT to Do

- Don't push from inside a worktree. `git push` is hard-blocked; merging is a
  human action via `dangeresque merge` + then `git push` on main.
- Don't close GitHub issues from a worker run. Humans close issues.
- Don't dispatch a second run on the same issue before merging + pushing the
  previous one.
- Don't widen scope beyond the Goal stated in the issue. Workers that widen
  scope cause review rejections.
- Don't touch `.dangeresque/*.md` or `.gitignore` from inside a worker run.
  Those are human-managed on main.
- Don't re-use a worktree name. Worktree creation hard-fails if the path exists.
- Don't read all prior runs — read only the newest under
  `.dangeresque/runs/issue-<N>/`.

## Pointers (details live elsewhere, in your project tree)

- `.dangeresque/AFK_WORKER_RULES.md` — full mode table, status language
- `.dangeresque/PERMISSIONS.md` — `acceptEdits`, `allowedTools`, `allow`
- `dangeresque --help` — full command surface
- `dangeresque stats --glossary` — result / verdict vocabulary

Generated by dangeresque v<X.Y.Z>.
```

## Observations

- The `dangeresque-create-issue` skill (skills/dangeresque-create-issue/SKILL.md) already encodes the issue body shape for interactive LLMs. The brief should reference the same template so the interactive-and-AFK story is consistent. If the skill ever drifts from the brief's template, that's a bug — worth filing a small follow-up issue in that event.
- `.dangeresque/CLAUDE.md.sample` already ships a short "Dangeresque Workflow" section (lines 22–37). It's a subset of what the brief will contain. After the brief ships, the CLAUDE.md.sample could be trimmed to just "See `dangeresque brief`" — that's a small follow-up, not part of this issue, but worth noting to avoid drift.
- `src/init.ts:80` correctly notes run artifacts are tracked in git. The brief should reiterate this — a fresh LLM might otherwise assume `.dangeresque/runs/` is gitignored like other artifact dirs.
- The `allow` subcommand (src/cli.ts:925–1000) is recent and well-documented in PERMISSIONS.md. Brief should show `dangeresque allow mcp` / `dangeresque allow bash "<pattern>"` as the canonical way to widen tool access, not raw `config.json` edits.

## Risks / Uncertainty

- **Content drift.** Workflow prose now exists in: README.md, CLAUDE.md.sample, AFK_WORKER_RULES.md, worker-prompt.md, and (soon) `src/brief.ts`. A change to the workflow must propagate to all of them. Mitigation: in `src/brief.ts` add a top-of-file comment listing the sibling sources that need the same update; the reviewer pass will flag mismatch if the update is partial.
- **Versioning the paste.** If a user pastes `dangeresque brief` into `CLAUDE.md` once and never refreshes, the embedded copy goes stale. The version footer makes staleness visible, but does not auto-refresh. An optional future follow-up: `dangeresque init` detects an older brief by marker comment and offers to update it. Not for this issue.
- **80-line constraint is tight.** At 80 lines, the "what NOT to do" section would shrink to 10 bullets and lose explanatory framing. The issue's criterion is "self-contained" — 130 lines better meets that without blowing up context. Recommend relaxing the 80-line mental model to "as short as possible while still self-contained" — about 130.
- **Auto-append to CLAUDE.md rejected.** If the maintainer wants auto-append anyway, it would need: marker comments to detect prior inserts, idempotent dedupe on re-run, and a way to handle user edits inside the marked region. None of that is hard, but it's more surface area than the opt-in pointer. Recommendation stands: opt-in.
