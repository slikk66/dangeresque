# Issue #35 — codex: full prompt visible in `ps` argv

## Status
investigating

## Summary

Hypothesis is **confirmed**. The codex worker and review paths assemble the entire prompt (worker-prompt.md content + task prompt with issue body + staged comments + absolute archive path + effort hint) into a single string and pass it as the final positional argument to `codex exec`. Node's `child_process.spawn` forwards those args as argv to the native codex binary, where `ps -p <pid> -o command` (or `ps -ef`) exposes them to any process on the host.

A clean fix exists with **no new runtime dependencies**: `codex exec` supports reading the PROMPT from stdin, via either passing `-` as the positional arg or omitting it. Dangeresque already pipes stdin to /dev/null via `stdio: ["ignore", ...]`; switching to `"pipe"` lets the parent write the prompt to `child.stdin` and close it.

## Confirmed Current Code Path (codex leaks)

`src/runner.ts:416-434` — `buildCodexWorkerArgs`:
```
prompt = workerPromptContent + "\n\n" + buildTaskPrompt(opts, archivePath)
       + "\n\nEffort preference: ${opts.config.effort} ...";
return ["exec","--json","--full-auto","--model",…,"--cd",worktreePath, prompt];
```

`src/runner.ts:436-471` — `buildCodexReviewArgs`: identical shape, concatenates `reviewPromptContent + buildReviewPrompt(...) + effort` into the last positional arg.

`src/runner.ts:570-574` (worker spawn) and `src/runner.ts:675-679` (review spawn):
```
spawn("codex", args, { cwd: worktreePath, stdio: ["ignore", "pipe", "pipe"], env });
```

Because `args[args.length-1]` is the full prompt, every byte appears in argv. `stdio: ["ignore", ...]` closes stdin (maps to /dev/null on Unix), so even if codex tried to read stdin, it would get immediate EOF.

The claim in the issue that Claude fully avoids this is **partially correct**: Claude uses `--append-system-prompt-file` for `config.workerPrompt` (line 319) so the large prompt template file contents do NOT appear in argv — only the path does. However, `buildTaskPrompt(opts, archivePath)` (line 332 and 387) IS passed positionally to Claude too, so Claude argv still contains the issue body + comments + paths. The codex exposure is strictly larger (it adds the full worker-prompt.md contents + effort hint on top of the same task prompt).

## Investigation Questions

### 1. Does `codex exec` support reading the prompt from stdin?

**Yes.** Source: `codex-rs/exec/src/cli.rs` defines:

```rust
#[arg(value_name = "PROMPT", value_hint = clap::ValueHint::Other)]
pub prompt: Option<String>,
```

With documented behavior: *"Initial instructions for the agent. If not provided as an argument (or if `-` is used), instructions are read from stdin."*

Implementation in `codex-rs/exec/src/lib.rs`:

```rust
fn resolve_prompt(prompt_arg: Option<String>) -> String {
    match prompt_arg {
        Some(p) if p != "-" => p,
        maybe_dash => {
            let behavior = if matches!(maybe_dash.as_deref(), Some("-")) {
                StdinPromptBehavior::Forced
            } else {
                StdinPromptBehavior::RequiredIfPiped
            };
            ...
        }
    }
}

fn read_prompt_from_stdin(behavior: StdinPromptBehavior) -> Option<String> {
    let stdin_is_terminal = std::io::stdin().is_terminal();
    // ... behavior checks (error if Forced and tty; skip if Piped-required and tty) ...
    let mut bytes = Vec::new();
    if let Err(e) = std::io::stdin().read_to_end(&mut bytes) {
        eprintln!("Failed to read prompt from stdin: {e}");
        std::process::exit(1);
    }
    // UTF-8 / UTF-16 LE/BE decoding ...
}
```

Two modes:
- `Some("-")` → `Forced`: always reads stdin; errors if stdin is a TTY.
- `None` → `RequiredIfPiped`: reads stdin only when NOT a TTY; otherwise would be empty and codex would fail fast.

Either mode works for dangeresque because Node's `child_process.spawn` with `stdio: ["pipe", ...]` gives codex a non-TTY pipe. Passing `-` explicitly is self-documenting and makes TTY-accidents error loudly rather than silently producing an empty prompt; **recommended**.

### 2. Does codex support `--prompt-file` or equivalent?

**No.** Full flag list from `codex-rs/exec/src/cli.rs`:

| flag | type | notes |
|---|---|---|
| `--skip-git-repo-check` | bool | — |
| `--ephemeral` | bool | — |
| `--ignore-user-config` | bool | — |
| `--ignore-rules` | bool | — |
| `--output-schema` | PathBuf | JSON Schema for response shape (not prompt) |
| `--color` | enum | — |
| `--json` / `--experimental-json` | bool | — |
| `--output-last-message`/`-o` | PathBuf | where to write final agent message |
| `PROMPT` | positional `Option<String>` | supports `-` / omitted → stdin |

No `--prompt-file`, `--input-file`, `--from-file`, or equivalent. Stdin is the only non-argv delivery path.

### 3. Temp-file fallback needed?

**No.** Stdin is strictly better than a temp file:
- No cleanup lifecycle; prompt lives only in-process memory bridging node and codex.
- No filesystem-readable leak (other users on the host could read a temp file with loose perms; the `ps` leak is the whole point).
- No new path-tracking state in `runner.ts` or in `.dangeresque/` hygiene.

Stdin approach wins on all three axes.

## Proposed Fix (for IMPLEMENT mode)

Scope: `src/runner.ts` only. Two build functions + two spawn sites.

1. **Refactor `buildCodexWorkerArgs`** to return `{ args: string[]; prompt: string }`:
   - Args end with `"-"` instead of the prompt string.
   - `prompt` carries the same concatenation as today.

2. **Refactor `buildCodexReviewArgs`** to return `{ args: string[]; prompt: string }` with the same shape.

3. **Worker spawn (`runWorker` codex branch, ~line 570)**:
   - Change `stdio: ["ignore", "pipe", "pipe"]` → `stdio: ["pipe", "pipe", "pipe"]`.
   - After `spawn(...)`, call `child.stdin.write(prompt); child.stdin.end();`.
   - Handle potential `EPIPE` on `stdin.write` by attaching an `error` listener that logs and proceeds (the exit-code path already handles downstream failure).

4. **Review spawn (`runReview` codex branch, ~line 675)**: identical change.

No changes to: `buildTaskPrompt`, `buildReviewPrompt`, `writeCodexRulesFile`, `commitWorkerChanges`, PID tracking, log capture, artifact path logic, or the claude paths.

## Risks / Uncertainty

- **EOF signaling.** Codex's `read_to_end` will block until stdin closes. Forgetting `child.stdin.end()` after `write` would hang the worker. Mitigation: call `end()` in the same tick as `write()`; no reason to hold stdin open.
- **TTY detection.** `std::io::stdin().is_terminal()` returns false for a piped fd, which is what Node's `"pipe"` produces. But if someone later changed dangeresque to `stdio: "inherit"`, codex would see a TTY and — with `-` — error loudly. Acceptable failure mode.
- **Large prompts + pipe backpressure.** Codex reads to EOF after its own setup completes, so there's a brief window where stdin buffer fills. Node streams handle this with the default highWaterMark; `write` returns false on backpressure. For a ~10KB prompt this is irrelevant but we should not assume unbounded. Current concat is bounded by issue body + comments + two prompt template files — tens of KB at most.
- **Not fully verified end-to-end.** I did not spawn a codex process under the proposed change in this run (INVESTIGATE mode). The TTY/pipe behavior is inferred from codex source + Node docs, which both are well-specified, but a live codex run with `ps` inspection should be the final proof under VERIFY.
- **codex exec version skew.** Stdin support comes from upstream main branch source I read. If the locally installed codex version (not queryable here — `codex --version` is blocked by the AFK sandbox permissions) predates that feature, stdin mode would fail. Low risk — OpenAI codex has supported this for a while — but IMPLEMENT should `codex --version` check before committing.
