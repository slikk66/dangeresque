## Summary

Root cause confirmed. The auto-follow behavior is driven by a single boolean-or at `src/cli.ts:514`:

```ts
const follow = followFlag || target.running;
```

`followFlag` is the explicit `-f`/`--follow` args parse at line 473. `target.running` comes from `listWorktrees()` → `readPidState()` which pings the worker PID. When the worker is alive, `follow` becomes `true` regardless of user intent, and `tailLog(…, { follow: true })` installs a `fs.watch` loop that only terminates when the worker PID dies or the user hits Ctrl-C.

Issue hint says "around line 422" — that's stale; actual code is at line 514. Not a blocker.

## Investigation Findings

### Call flow
1. `cmdLogs(args)` at `src/cli.ts:462` parses `-f`/`--follow` into `followFlag` (line 473).
2. Resolves target worktree; reads PID file; decides `phase` (worker vs review).
3. Line 514: `const follow = followFlag || target.running;` ← the problem.
4. Line 515-520: invokes `tailLog({ sessionPath, follow, raw, pid })`.

### What `tailLog` does with `follow` (src/logs.ts:184-243)
- Always calls `await printFile(sessionPath, raw)` first — the snapshot.
- `if (!follow) return;` at line 194 — early exit is already correctly implemented.
- Otherwise opens `fs.watch`, streams growth, polls the PID every 3s. Terminates when PID dies; otherwise Ctrl-C.

So `tailLog` itself is well-behaved. The bug is purely the `follow` computation in `cmdLogs`. No changes needed in `logs.ts`.

### `--raw` and `--review` are orthogonal
- `--raw` only affects formatting inside `printFile` / the watcher loop. Zero interaction with `follow`.
- `--review` selects which phase log to read (worker vs review). Zero interaction with `follow`.
- Therefore: dropping `|| target.running` automatically gives `--raw` snapshot-and-exit too. Satisfies VC #5 with no extra code.

### Exit code
`cmdLogs` has no explicit `process.exit(0)` on the success path. When `tailLog` returns, `main()` resolves, Node exits 0 naturally. ✓ satisfies VC #1 ("exits 0").

### Stderr header today
`cli.ts:510-512` already prints one line to stderr:
```ts
console.error(
  `Branch: ${target.branch}  Phase: ${phase}  ${target.running ? "RUNNING" : "IDLE"}`,
);
```
This is a pre-print header, not a footer. It stays. The discoverability hint should be a separate stderr message emitted AFTER `tailLog` returns (footer), gated on `target.running && !followFlag`. Placing it after means it doesn't appear when the user actually tails (follow=true, tailLog blocks until EOS).
