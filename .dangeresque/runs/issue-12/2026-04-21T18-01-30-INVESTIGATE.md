## Goal (restated)

Add `--issue-fixture <path>` to `dangeresque run`. Reads an `IssueData`-shaped JSON from disk in place of `gh issue view`, and avoids calling `gh` entirely (no fetch, no result comment). Must coexist cleanly with existing `--issue <N>` and ad-hoc (`NEXT_TASK.md`) flows.

## Findings

### 1. How `--issue <N>` flows today

- **Parse** — `src/cli.ts:178-186` reads `--issue <N>` into `issueNumber`. No other flags currently overlap.
- **Fetch** — `src/cli.ts:190-204` calls `fetchIssue(projectRoot, issueNumber)` which is defined at `src/runner.ts:70-91`. That runs `gh issue view <N> --json title,body,comments` via `execSync` and returns `IssueData`.
- **Pass into worker** — `src/cli.ts:230-236` forwards `issueData` into `runWorker`. In `buildTaskPrompt` (`src/runner.ts:111-134`) the prompt is assembled from `issueData.number`, `issueData.title`, `issueData.body`, plus `formatIssueComments()` output. **Engine-agnostic**: codex and claude both go through `buildTaskPrompt` (`src/runner.ts:182, 265`).
- **Archive subdir** — `computeRunArchivePath` at `src/runner.ts:42-50` uses `opts.issueData?.number`. If a number is present, runs are filed under `.dangeresque/runs/issue-<N>/`. If not, under `adhoc/`. A fixture supplies a number, so fixture runs will land under `issue-<N>/` automatically — **no change required**.
- **Scope check** — `src/cli.ts:296-329` reads `issueData.body` and flags files changed outside what the body mentions. Fixtures retain this behavior (they have a body).
- **Post-run gh comment** — `src/cli.ts:270-288` (failure path) and `src/cli.ts:380-396` (success path) both call `postRunComment`, each gated on `if (issueNumber)`. These are the ONLY `gh` calls in the main run path beyond `fetchIssue`.
- **Artifact** — `src/cli.ts:239-250` builds the artifact with `issueNumber`. Inside `ArtifactBuilder.build()` (`src/artifact.ts:192-218`):
  - `issue_number: init.issueNumber ?? null`
  - `issue_url: buildIssueUrl(init.projectRoot, init.issueNumber)` — the URL is computed from `git config --get remote.origin.url` + the number (`src/artifact.ts:321-335`).

### 2. Existing ad-hoc flow (no `--issue`)

- `src/cli.ts:190` leaves `issueData` undefined.
- `buildTaskPrompt` (`src/runner.ts:136-146`) reads `NEXT_TASK.md` and instructs the worker to read it in full. Archive lands under `adhoc/`.
- No `gh` calls, no `postRunComment` call (gated on `issueNumber`). This path already demonstrates the "skip gh" behavior at half-strength; fixtures should slot in alongside it.

### 3. What the fixture path must change

- **CLI parsing**: accept `--issue-fixture <path>`. Error + non-zero exit if combined with `--issue`.
- **Fixture loader**: new `loadIssueFixture(path: string): IssueData` in `src/runner.ts`. Reads file, `JSON.parse`, validates shape: `number: number`, `title: string`, `body: string`, `comments: Array<{body,author:{login},isMinimized}>`. Throws with a clear error on missing file / malformed JSON / wrong shape. Stdlib only (`node:fs.readFileSync`).
- **Route fixture through existing plumbing**: after loading, set `issueData = <fixture>` and `issueNumber = issueData.number`. Everything downstream (`runWorker`, prompt building, scope check, archive path, builder.issueNumber) works unchanged.
- **Skip gh side-effects**: gate BOTH `postRunComment` call sites on "is fixture?" in addition to `if (issueNumber)`. Simplest shape: a `fixtureUsed: boolean` local in `cmdRun`.
- **Force `issue_url: null` in the artifact**: `buildIssueUrl` would currently yield a URL for any number + valid remote. The issue's verification criterion requires `issue_url: null` for fixture runs. Cleanest fix: add one field to `BuilderInit` in `src/artifact.ts` that suppresses URL lookup. Two viable shapes (pick one in IMPLEMENT):
  - **Option A** (preferred): `issueUrl?: string | null` override. When `init.issueUrl !== undefined`, use it verbatim (including explicit `null`); otherwise fall back to `buildIssueUrl(...)`. Most general.
  - **Option B**: `fromFixture?: boolean`. When true, `issue_url` is forced to null. More self-documenting but single-purpose.
  Either option is additive and does NOT change the `RunArtifact` schema — so `ARTIFACT_SCHEMA_VERSION` stays at `"1"`.

### 4. Public API surface

- `src/index.ts:11-19` currently re-exports `fetchIssue`, `postRunComment`, `runWorker`, `runReview`, and `IssueData` from `./runner.js`. Adding `loadIssueFixture` to the same re-export block keeps the helpers grouped and is consistent with the issue's "re-export if public" suggestion. Recommend yes — it's a peer of `fetchIssue`.

### 5. Help text & UX

- Mutual exclusion error message should name both flags and exit non-zero — a `console.error` + `process.exit(1)` matches the pattern used for `--issue` parse failures (`src/cli.ts:180-183`).

## Risks / Uncertainty / Observations

- **Worktree naming collision**: auto-name is `<mode>-<issueNumber>` (`src/cli.ts:213-215`). A fixture with `number: 999` and a real issue #999 would generate the same worktree path. Creation hard-fails when the path already exists (`src/runner.ts:324-333`), so the second run gets a clean error — not silent data loss. Users can override with `--name`. Note, don't fix.
- **Scope-check noise for fixtures**: `src/cli.ts:296-329` flags any changed file whose path isn't in `issueData.body`. Fixture bodies written for offline smoke tests typically won't list file paths, so every changed file will produce a warning. Harmless (warning only, doesn't fail the run), but worth knowing. Not in scope to change.
- **Artifact indistinguishability**: consumers of the run artifact JSON can't tell a fixture run from a real-issue run with a broken remote — both yield `issue_url: null`. If telling them apart matters later, an additive `fromFixture: boolean` field on `RunArtifact` would require an `ARTIFACT_SCHEMA_VERSION` bump per `.claude/CLAUDE.md`. The current issue does not ask for this; do not add.
- **File read errors**: a relative fixture path resolves against `process.cwd()`. Document / recommend absolute paths in error messages if the IMPLEMENT run hits this. Not blocking.
- **Validation strictness**: minimal shape validation (type-check `number`, `title`, `body`, `comments`) is sufficient — `comments` items are consumed by `formatIssueComments` which already gracefully handles `isMinimized` via `filter`. Extra fields in the fixture are ignored (spread-safe). No need for JSON Schema.
