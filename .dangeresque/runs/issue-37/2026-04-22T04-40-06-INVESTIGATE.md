## Summary

Hypothesis CONFIRMED. `mergeWorktree` wraps the git-merge and all post-merge cleanup in a single try/catch, so any cleanup failure (worktree removal, branch deletion) surfaces under the `Merge failed:` banner even after `git merge` succeeded and main has advanced.

## Root cause

Single try/catch at `src/worktree.ts:290-338` spans three distinct git operations (`git merge`, `git worktree remove`, `git branch -d`). Any throw from any of them produces `"Merge failed: ${err.message}"`, regardless of which phase threw. The state information required to distinguish (`headBefore`, `headAfter`) is captured within the try but not consulted in the catch.

## Proposed fix shape (for IMPLEMENT, not done here)

Match the issue's proposal: explicit per-phase try/catches with per-phase return payloads. Suggested shape consistent with the current interface:

- Extend the return type to `{ success: boolean; message: string; phase?: "merge" | "cleanup" | "branch-delete"; headAdvanced?: boolean; headBefore?: string; headAfter?: string; }`.
- Move `headBefore` capture to the top (before any try).
- Wrap `git merge` in its own try; on failure, return `{ success: false, phase: "merge", headAdvanced: false, message: "Merge did not occur: …. Main is unchanged." }`.
- Capture `headAfter`; keep the existing no-op branch.
- Wrap `git worktree remove` in its own try; on failure, return `{ success: false, phase: "cleanup", headAdvanced: true, message: "Merge succeeded — main is now at <sha>. Worktree cleanup failed: …. Recovery: (1) inspect <path>, (2) 'git worktree remove --force <path>', (3) 'git branch -d <branch>'." }`.
- Make the branch-delete phase explicit too (stop swallowing silently; treat a genuine failure as `phase: "branch-delete"` with recovery guidance, but keep a narrow tolerance for the "branch already gone" case if that's really observed in practice).
- `cmdMerge` continues to print `result.message` verbatim — the honesty lives inside the message string, so callers don't need to change semantics. The phase/`headAdvanced` fields become useful for future programmatic callers (and eventually for `discardWorktree` to mirror the shape).

For `discardWorktree`: same pattern, but "merge" phase is absent; phases are `"cleanup"` and `"branch-delete"`. Update the incorrect comment at L369.
