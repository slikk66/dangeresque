## Summary

Split `mergeWorktree` and `discardWorktree` into explicit per-phase try/catches and returned phase-tagged results so the user-facing message tells the truth about what happened. The former single try/catch in `mergeWorktree` wrapped three distinct git operations (`git merge`, `git worktree remove`, `git branch -d`) and surfaced every failure as `"Merge failed: â€¦"` â€” the exact conflation #37 reports. After this change, a post-merge cleanup failure reads `"Merge succeeded â€” main is now at <sha>. Worktree cleanup failed: â€¦. Recovery: â€¦"`, making main's advanced state explicit and giving concrete recovery commands.

The return type is now `WorktreeOpResult { success, message, phase?, headAdvanced?, headBefore?, headAfter? }`. New fields are optional/additive; `cmdMerge` and `cmdDiscard` continue to print `result.message` verbatim, so the public CLI behavior stays word-for-word identical in the happy path and only the error wording changes. `src/index.ts` re-exports the new type names.

The silent branch-delete swallow in both functions was removed per the plan. The incorrect "may already be deleted by worktree remove" justifying comments (factually wrong â€” `git worktree remove` does NOT delete the branch) were deleted along with the swallow.

## Risks / Uncertainty

- **Return shape**: `WorktreeOpResult` is strictly additive over the old `{ success, message }` anonymous shape. `cmdMerge`/`cmdDiscard` do not read the new fields. External consumers of `src/index.ts` would see `mergeWorktree`/`discardWorktree` returning a named type now, but `success` and `message` are still present with the same semantics.
- **Upstream-ref trick in the branch-delete test**: this is a synthetic way to force `-d` to fail, but it exercises the exact exception pathway from the `execSync(\`git branch -d â€¦\`)` call the real bug surfaces on. Not a true substitute for reproducing a ref-corruption or mid-call race, but sufficient to lock in the phase-detection contract.
- **Discard behavior change**: previously, a branch-delete failure after successful worktree-remove was silently ignored and reported as success. Now it returns `phase: "branch-delete"` with `success: false`. Any script checking only `exitcode == 0` from `dangeresque discard` will start seeing non-zero exits on the newly-surfaced failure mode. That's the intended correction per the issue (dual of the primary merge-conflation bug â€” partial completion reported as full success).
- **Stale DESIGN.md line numbers**: `docs/DESIGN.md:168` references `src/worktree.ts:244-297`; actual `mergeWorktree` now spans 298-415. Pre-existing drift; not caused by this change. Noted under Observations; not touched per scope discipline.
