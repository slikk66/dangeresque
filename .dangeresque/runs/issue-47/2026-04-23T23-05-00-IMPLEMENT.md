## Summary

Shipped the canonical/`.local.md` split for the three prompt files and gave `dangeresque init` real upgrade semantics layered on Phase 1's `readPromptWithLocal` plumbing. Added an `copyWithLocalOverlay` helper in `src/init.ts` that handles four cases: `created`, `upgraded`, `initialized-local`, `customized-warn`. The customized-warn path is non-destructive — user drives any reconciliation, no auto-migration.

Updated the worker startup sequence (in both `config-templates/worker-prompt.md` and `.dangeresque/worker-prompt.md` for self-dogfooding) to instruct the worker to Read `.dangeresque/AFK_WORKER_RULES.local.md` after the canonical. AFK_WORKER_RULES is Read-at-runtime (not system-prompt-injected), so the spawn-time concat helper from Phase 1 doesn't reach it — the worker prompt instruction is the equivalent mechanism.

Added 8 new tests: 6 unit tests covering each action of `copyWithLocalOverlay` plus a regression guard on `SPLIT_BASE_NAMES`, plus 2 integration tests that exercise `initProject` end-to-end against scratch project roots (the second test directly proves idempotency: a user-edited local file survives a re-init).

### `src/init.ts`

1. **New exports** — `SPLIT_BASE_NAMES` (the three split-set bases) and `copyWithLocalOverlay(templatesDir, configDir, baseName, warnings)` returning a `CopyAction` (`"created" | "upgraded" | "initialized-local" | "customized-warn"`).

2. **Helper logic** — four cases:
   - canonical missing → copy both (canonical + `.local.md` stub) from templates → `created`
   - canonical present, local present → overwrite canonical, leave local → `upgraded`
   - canonical present matches shipped, no local → copy local stub only → `initialized-local`
   - canonical diverges from shipped, no local → push warning, touch nothing → `customized-warn`

3. **Warning text** — multi-line, includes path to shipped canonical AND user file, instructs the user to extract their delta into `.local.md` and re-run init. No auto-migration.

4. **Routing in `initProject`** — the `readdirSync(templatesDir)` loop now skips `.local.md` files (handled by their canonical pair), special-cases `SPLIT_BASE_NAMES` to call `copyWithLocalOverlay`, and falls through to the existing skip-if-exists logic for everything else. `claude-settings.json` and `CLAUDE.md.sample` continue to be skipped here (Phase 3 handles their deprecation).

### `test/unit/init.test.ts` (NEW)

Eight tests using `mkdtempSync` per the runner.test.ts pattern:

- **Unit (synthetic templates dir)**:
  1. fresh project → `created` (canonical + local both written; assert content matches)
  2. canonical matches shipped, no local → `initialized-local` (local stub written, canonical re-written byte-identical)
  3. canonical + local both present → `upgraded` (canonical refreshed, local untouched)
  4. customized canonical, no local → `customized-warn` (warning text checked for "has been customized", "worker-prompt.local.md", "Re-run dangeresque init"; neither file touched, local NOT created)
  5. upgrade preserves arbitrary local content byte-for-byte (stress test with multi-line user content)
  6. `SPLIT_BASE_NAMES` regression guard — exact contents asserted
- **Integration (real `config-templates/` via `initProject`)**:
  7. fresh scratch project → all 6 split files written (3 canonical + 3 `.local.md`)
  8. second invocation idempotent → user-edited `worker-prompt.local.md` survives re-init byte-for-byte

Tests silence `console.log` during `initProject` calls to keep test output readable.

## Risks / Uncertainty

- **Customized-warn UX is the intended path for projects that hand-edited `.dangeresque/*.md`** before this phase shipped. Those users see the LOUD warning on next init, decide whether to migrate. No auto-fix; per INVESTIGATE Q4 option (b). If telemetry/feedback shows users routinely ignore the warning, Phase 3+ can revisit (option (c) — a `dangeresque migrate` subcommand).
- **`copyWithLocalOverlay` requires the matching `.local.md` source file in `config-templates/`.** If a packaging bug omits one (e.g. someone adds a new `SPLIT_BASE_NAMES` entry without shipping the stub), the helper will throw `ENOENT` on `copyFileSync(localSrc, …)`. That's loud-fail-by-design — packaging mistakes should not be silent. Noted as a constraint future contributors must respect when extending the split set.
- **Live CLI smoke not directly performed.** The sandbox blocked invoking `node dist/cli.js init` and `dangeresque init` directly with chained `cd` operations. The two integration tests that call `initProject` against a scratch directory cover the same ground programmatically — fresh install + idempotent re-init. The CLI thin-wraps `initProject`, so the only thing not exercised is argv parsing in `cli.ts`, which is unchanged in this phase.
- **`config.ts validateSetup` does not check for `.local.md` presence.** Open question carried from INVESTIGATE Q5; the staged plan kept it out of Phase 2. Currently a missing `.local.md` is silent — `readPromptWithLocal` returns canonical-only when the local is absent. Acceptable for v1; revisit if half-migrated projects start showing up in support traffic.
- **`.dangeresque.pid` appeared in git status.** Existing untracked file from a prior run; not committed. Not in scope to clean up.
