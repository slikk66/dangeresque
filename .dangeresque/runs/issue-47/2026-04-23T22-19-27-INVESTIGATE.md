## Investigation Questions — answers

### Q1. Composition mechanism

**Recommendation: concat at spawn.**

- **Model-driven read**: the worker would need a Read tool call to pick up `.local.md` content into context after the system prompt is already set. Costs one round-trip before any real work, and the content wouldn't be part of the system prompt proper — which matters for instruction priority. Reject.
- **Explicit include-marker** (e.g. `<!-- INCLUDE: … -->`): custom syntax, bespoke parser, foot-gun for users who copy-paste the marker elsewhere. Reject.
- **Concat at spawn**: engine-agnostic, zero new syntax, fits the existing codex path (already reading the file into a string), and adds one file-I/O call to the claude path. Accept.

**How it looks for each engine:**

- **Codex**: `readFileSync(canonical) + readFileSync(local, { flag: "optional" })` — currently concat is already the codex model. Two-line change in `buildCodexWorkerArgs` / `buildCodexReviewArgs`.
- **Claude**: two options, both viable:
  - **(a) Merged temp file**: read both, write `<worktree>/.dangeresque/.merged-<worker|review>-prompt.md`, pass that path via existing `--append-system-prompt-file`. Gitignored. Zero CLI-flag changes.
  - **(b) Direct flag switch**: replace `--append-system-prompt-file <path>` with `--append-system-prompt <content>` and pass the merged string. Argv-visible, but unlike Issue #35/#43 this content is system-prompt scaffolding, not user-owned issue body — **argv leak is not a concern here**.

I lean (b) — simpler, no temp-file cleanup, no new on-disk artifact. (a) is fine if we prefer symmetry with the existing file-based flow. Both are one-screen diffs in `buildClaudeWorkerArgs` / `buildClaudeReviewArgs`.

### Q4. Migration for existing customized files

The issue lists (a) detect-diff-and-prompt, (b) document + init warn, (c) `dangeresque migrate` subcommand.

**Recommendation: option (b) with an explicit opt-out path.** Here's the specific flow init would run for each split-pair `<name>.md` / `<name>.local.md`:

1. If `.local.md` exists → **normal upgrade path**: overwrite `.md` from canonical; leave `.local.md` untouched. Print `Refreshed <name>.md (canonical); <name>.local.md preserved`.
2. If `.local.md` does NOT exist AND existing `.md` == shipped canonical byte-for-byte → create an empty `.local.md` (with a single-line header comment: `<!-- Project-specific additions to <name>.md — dangeresque will never overwrite this file. -->`). Overwrite `.md` (no-op since identical). Print `Initialized <name>.local.md (empty — add project overrides here)`.
3. If `.local.md` does NOT exist AND existing `.md` differs from shipped canonical → **do NOT auto-move**. Print a LOUD warning:

   ```
   ⚠️  .dangeresque/<name>.md has been customized and does not match the shipped canonical.
       Your changes will not be lost — they're preserved in-place.
       To pick up upstream improvements:
         1. Move your customizations out of <name>.md into <name>.local.md
         2. Re-run dangeresque init to install fresh canonical <name>.md
       Or keep the current file as-is; upgrades will continue to skip it.

       Shipped canonical: <templates>/<name>.md
       Your file:         .dangeresque/<name>.md
   ```

   Do not copy the canonical to any other path; do not touch `.md`. User drives the migration.

Why not (a) detect-diff-and-prompt: init is non-interactive in CI/AFK usage; prompts would hang or make bad defaults. The warning form in (b) surfaces the problem without locking up.

Why not (c) a separate `migrate` subcommand: adds CLI surface for a one-time flow. The init path already runs on every upgrade attempt, so embedding the detection there catches every user automatically. If (b)'s warning turns out to be insufficient in practice, (c) can be added later as a helper (`dangeresque migrate --file worker-prompt.md` that extracts the diff into `.local.md` automatically).

**Edge case**: what about users who DELETED sections from canonical? A user-local file can only **append**, not override/remove. If a user wants to remove a canonical rule, they'd have to modify `.md` (which upgrades would then overwrite) OR we'd need a richer merge model. Scope out for v1: document that `.local.md` is additive-only. Users who want to remove canonical rules must fork the prompt file.

### Q6. Spawn-time concat — where

`src/runner.ts`. Add one helper:

```typescript
/**
 * Read <configDir>/<baseName> and, if present, <configDir>/<baseName>.local.md,
 * returning the concatenated content (canonical first, then "\n\n", then local).
 */
export function readPromptWithLocal(configDir: string, baseName: string): string {
  const canonical = readFileSync(join(configDir, baseName), "utf-8");
  const localName = baseName.replace(/\.md$/, ".local.md");
  const localPath = join(configDir, localName);
  if (existsSync(localPath)) {
    const local = readFileSync(localPath, "utf-8").trim();
    if (local.length > 0) return canonical + "\n\n" + local;
  }
  return canonical;
}
```

## Risks / Uncertainty

- **`--append-system-prompt-file` repeatability unknown.** Could not test (`claude --append-system-prompt-file` needs a valid follow-arg and I can't run claude interactively in this sandbox). If it's repeatable, passing two `--append-system-prompt-file` flags (one canonical, one local) avoids ever concatenating in-memory. If not, we concat at spawn (recommended path). Either way lands the same user-visible behavior.

- **CLAUDE.md auto-load semantics across Claude versions.** Claude Code auto-loads `CLAUDE.md` at session start. The pointer line approach assumes the worker will Read the referenced `DANGERESQUE.md` when prompted. This is more a convention than a guarantee — it's worked fine for `.dangeresque/AFK_WORKER_RULES.md` so far (it's just a Read target), but if a future Claude version changes auto-load behavior, re-evaluate.

- **Worker-prompt wording change for AFK_WORKER_RULES**: the worker-prompt canonical (shipped) tells the worker to "Read `.dangeresque/AFK_WORKER_RULES.md`". For the split to reach the worker, update that to "Read `.dangeresque/AFK_WORKER_RULES.md`, then Read `.dangeresque/AFK_WORKER_RULES.local.md` if it exists." This is a canonical-side edit (in `config-templates/worker-prompt.md`), so post-v1 it propagates automatically. But the v1 deploy is a chicken-and-egg: existing projects need to reinit to pick up the new instruction, AND the new canonical needs the canonical/local template split installed. Sequence matters in the implementation PR.

- **Deprecating `CLAUDE.md.sample` cleanly**: some users may have already copied `CLAUDE.md.sample` content into their `CLAUDE.md` and customized it. Removing the `.sample` file is low-risk (it's a reference they already consumed), but `src/init.ts:69` has `if (file === "claude-settings.json" || file === "CLAUDE.md.sample") continue;` — that special-case block changes. Also `src/init.ts:135-146` drives the bare-existence check against CLAUDE.md.sample. Both spots need coordinated edits.

- **Validation setup checks (`src/config.ts:114-123`)**: today checks `workerPrompt` and `reviewPrompt` existence. Should it also verify the canonical vs `.local.md` pair? If `.local.md` is missing but canonical exists, is that a warning or an error? My take: don't error (keeps backward-compat for projects mid-migration); print a soft hint via init but don't block runs.

- **Re-running `dangeresque init` on a project that had only canonical files and customized them**: the warn-don't-auto-move approach from Q4 puts the upgrade burden on the user. For the first version, this is safer than surgery. If dogfood shows users routinely ignore the warning and keep accumulating drift, reconsider after measuring.
