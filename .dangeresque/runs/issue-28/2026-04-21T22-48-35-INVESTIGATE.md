## Evidence

### Canonical result classifications

`src/artifact.ts:9` defines:

- `success`
- `partial_success`
- `failure`

Actual derivation rules in `deriveResult()`:

- `failure`: worker exit code is non-zero, or the run result archive is missing (`src/artifact.ts:321-322`).
- If review ran and review process exit code is non-zero: `partial_success` (`src/artifact.ts:324-328`).
- If review ran and reviewer verdict is `reject`: `failure` (`src/artifact.ts:331`).
- If review ran and reviewer verdict is `accept`: `success` (`src/artifact.ts:332`).
- If review ran and verdict is anything else, including `needs_human_review` or `unknown`: `partial_success` (`src/artifact.ts:333`).
- If review did not run, result is `partial_success` when `scopeViolations.length > 0`, otherwise `success` (`src/artifact.ts:336`).

Suggested wording:

- `success`: Worker completed, produced its run artifact, and either the reviewer accepted it or no review ran and no scope violations were recorded.
- `partial_success`: Worker completed and produced its run artifact, but the run needs attention because review errored, review returned no accepting verdict, or review was skipped while scope violations were recorded.
- `failure`: Worker failed, the run artifact was missing, or the reviewer explicitly rejected the run.

### Reviewer verdict values

`src/artifact.ts:11-16` defines:

- `accept`
- `reject`
- `needs_human_review`
- `skipped`
- `unknown`

Actual derivation rules:

- `accept`, `reject`, and `needs_human_review` are parsed from run result markdown matching `**Verdict:** ACCEPT`, `REJECT`, or `NEEDS_HUMAN_REVIEW` / equivalent spacing and hyphen variants (`src/artifact.ts:255-264`).
- If worker exit code is non-zero, reviewer verdict is `unknown` (`src/artifact.ts:272`).
- If no review phase exists or the review phase is marked skipped, reviewer verdict is `skipped` (`src/artifact.ts:273-274`).
- If the archive does not exist, cannot be read, or contains no parseable verdict, reviewer verdict is `unknown` (`src/artifact.ts:275-280`).

Suggested wording:

- `accept`: The reviewer found the worker's changes acceptable.
- `reject`: The reviewer rejected the worker's changes; this makes the run a `failure`.
- `needs_human_review`: The reviewer did not accept or reject outright and asks for human judgment; this makes the run `partial_success`.
- `skipped`: No review verdict was produced because review was intentionally skipped.
- `unknown`: Dangeresque could not derive a reviewer verdict, usually because the worker failed, the artifact was missing/unreadable, or the markdown verdict was absent/unparseable.

### Scope violations

Relevant code:

- Post-worker scope check computes changed files via `git diff main...HEAD --name-only`, excludes `.dangeresque/runs/`, and flags files whose path is not included in the issue body plus selected issue comments (`src/cli.ts:363-389`).
- `scope_violation` is a `FailureCategory` value (`src/artifact.ts:18-25`).
- `scope_violation` contributes to `failure_categories` only when it actually caused a downgrade: worker succeeded, archive exists, reviewer did not run, and scope violations were recorded (`src/artifact.ts:298-306`).
- When review ran, reviewer verdict is authority on scope; `docs/DESIGN.md:243-254` explains that `scope_violations[]` remains diagnostic but does not downgrade an accepted reviewed run.

Suggested wording:

- `scope_violations[]`: Changed files that were not mentioned in the issue body or selected issue comments, excluding run artifacts.
- `scope_violation`: Failure category emitted only when those recorded scope violations downgraded a run because review was skipped; when review ran, reviewer verdict controls the result.

### Review run vs auto-skip

Relevant code:

- `--no-review` sets `review = false` during run option parsing (`src/cli.ts:179-180`).
- Review is auto-skipped for `INVESTIGATE` and `VERIFY` by `SKIP_REVIEW_MODES = new Set(["INVESTIGATE", "VERIFY"])` (`src/cli.ts:421-426`).
- If review is disabled with `--no-review`, artifact builder records skip reason `--no-review` (`src/cli.ts:440-441`).
- `docs/DESIGN.md:126-129` documents that review is skipped for modes that by definition produce no code changes: INVESTIGATE and VERIFY.

Suggested wording:

- Review normally runs after successful worker runs for code-changing modes.
- Review is automatically skipped for `INVESTIGATE` and `VERIFY`.
- Review is also skipped when the run uses `--no-review`.
- Skipped review appears as reviewer verdict `skipped`, with a skip reason in the review phase.

### Stats output consumers

`src/stats.ts` does not derive meanings. It aggregates existing artifact fields:

- `computeStats()` counts `a.result` and `a.reviewer_verdict` directly (`src/stats.ts:155-158`).
- `formatStats()` prints "Results:", "Reviewer verdicts:", and "Failure categories:" without definitions (`src/stats.ts:194-219`, `src/stats.ts:264-271`).

This supports the issue rationale: readers of `dangeresque stats` need a glossary because stats output intentionally summarizes raw artifact values.
