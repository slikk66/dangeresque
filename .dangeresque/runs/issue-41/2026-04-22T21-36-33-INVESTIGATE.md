## Summary

Two separable defects in one report. Both are confirmed.

**Defect 1 — `mcp__*` is not a valid claude-code permission pattern.** The DEFAULT_CONFIG at `src/config.ts:62` ships `"mcp__*"` as the catch-all MCP allow rule. Per the current Anthropic permissions documentation (https://code.claude.com/docs/en/permissions, §"MCP"), the only documented forms are:
- `mcp__<server>` — matches any tool from that server
- `mcp__<server>__*` — matches any tool from that server (wildcard variant)
- `mcp__<server>__<tool>` — matches one specific tool

The bare `mcp__*` form (wildcard at the server-name slot) is **not in the documented list**. The upstream issue the maintainer cited (anthropics/claude-code#3107, closed 2025-07-07 by an Anthropic contributor) explicitly states "Permission rules do not support wildcards" at that position and points users to the same docs. The user's failing tool was `mcp__plugin_context7_context7__query-docs` — server name `plugin_context7_context7` (a plugin-installed MCP server, prefixed with `plugin_`). Even the narrow form would have to be `mcp__plugin_context7_context7` or `mcp__plugin_context7_context7__*`.

**Defect 2 — a new user hitting "tool was blocked" has no in-product path to self-serve.** Three independent gaps:
- `dangeresque --help` (built by `usageForEngine` at `src/cli.ts:39-87`) does not mention `allowedTools`, `disallowedTools`, `.dangeresque/config.json`, or the word "permission" anywhere.
- `dangeresque init` (`src/init.ts`) ships `AFK_WORKER_RULES.md`, `worker-prompt.md`, `review-prompt.md`, and `CLAUDE.md.sample` — but no `config.json` template, no `PERMISSIONS.md`, no `TROUBLESHOOTING.md`. So the user's `.dangeresque/` directory after `init` contains zero hint that `config.json` exists or what shape it takes.
- README does cover it (lines 411-437) but it's buried below the entire workflow walkthrough, and the example in the "Default Tool Permissions" block at `README.md:432` is itself wrong: `mcp__*` (all MCP servers)`. The README is teaching the broken pattern.
- When a tool is blocked, `src/logs.ts::formatClaudeLine` renders the raw `is_error: true` tool_result as a red `[error]` line (line 105). The text "Claude requested permissions to use ... but you haven't granted it yet" is Claude's verbatim error string. Dangeresque adds zero context — no "tip: add `mcp__<server>` to allowedTools in `.dangeresque/config.json`" hint, no link to docs.

## Investigation Answers

### Q1 — Confirm matcher behavior for `mcp__*` vs `mcp__<server>__*` (as of 2026-04)

**Answer.** Verified against the live Anthropic permissions doc on 2026-04-22 (https://code.claude.com/docs/en/permissions §"MCP"). Quoted verbatim:

> ### MCP
> * `mcp__puppeteer` matches any tool provided by the `puppeteer` server (name configured in Claude Code)
> * `mcp__puppeteer__*` wildcard syntax that also matches all tools from the `puppeteer` server
> * `mcp__puppeteer__puppeteer_navigate` matches the `puppeteer_navigate` tool provided by the `puppeteer` server

Bare `mcp__*` is absent from this enumeration. Cross-checked against upstream issue anthropics/claude-code#3107 (closed COMPLETED 2025-07-07): contributor `rboyce-ant` wrote "The correct syntax here is `mcp__atlassian`. Permission rules do not support wildcards" — referring to the server-name slot. The reporter's `mcp__atlassian__*` form was treated by claude as an unmatched literal, which is consistent with what we saw for `mcp__plugin_context7_context7__query-docs`.

That said: the docs DO show `mcp__puppeteer__*` as a valid wildcard in the *tool-name* slot. So a wildcard at the third `__` position works; a wildcard at the second (server-name) position does not.

**Recommendation: drop `mcp__*` from DEFAULT_CONFIG.** Replace with explicit guidance to add `mcp__<server>` per server. Keeping `mcp__*` is worse than removing it — it pattern-matches nothing reliably, gives users a false sense of "MCP is allowed by default," and causes silent block-and-prompt failures in headless `-p` mode (where there is no human to prompt, so the tool just errors out).

### Q2 — UX fix so a new user can self-serve

The maintainer proposed three candidates. Honest assessment of each:

**(a) Permissions section in `dangeresque --help`.** Cheapest. The help string is already engine-aware (`usageForEngine` at `src/cli.ts:39-87`) and has natural room under "Run options" or as a separate "Permissions" block. A 6-line addition mentioning `.dangeresque/config.json`, the `allowedTools` key, and a one-line example like `"mcp__<server>", "Bash(npm install *)"` would solve 80% of "where do I even start" cases. Only downside: users who never run `--help` won't see it.

**(b) `TROUBLESHOOTING.md` shipped by `dangeresque init` into `.dangeresque/`.** Highest-discoverability for the right audience: the user who's already looking at `.dangeresque/` after a failure. Costs one new file in `config-templates/` and one entry in init's copy loop (init.ts already iterates the templates dir at `src/init.ts:68`, so adding the file is zero new code). Should be ~30 lines covering: how to inspect `dangeresque logs <branch>` for `[error]` lines, the JSON shape of `config.json`, `allowedTools` syntax with one MCP example and one Bash example, the `mcp__<server>` form per Anthropic docs (with link), and a note that `acceptEdits` does NOT auto-approve MCP or arbitrary Bash.

**(c) Informative log line when a tool is blocked.** Highest in-context value (the user is staring at the error when this fires) but the most invasive. Implementation: `src/logs.ts::formatClaudeLine` already detects `block.is_error === true`; it could regex-match on "requested permissions to use" and append a hint pointing at `.dangeresque/config.json`. This is a runtime-only enhancement (no behavior change to the worker, no code-shape change to the artifact). Risk: false positives — Claude returns lots of `is_error` tool_results, not all of them are permission failures. Mitigation: anchor on the literal "requested permissions to use" prefix, which is a stable claude-code error string today.

**My recommendation: do all three, in this priority order: (b) → (a) → (c).** They compose: (b) is the durable reference, (a) gets the user there from their first `--help`, (c) catches users who never read either. They are independent slices; each can ship as its own IMPLEMENT issue.

### Q3 — Should `dangeresque init` dump a "common additions" commented block into `.dangeresque/config.json`?

**Cannot literally do this in JSON** — JSON does not support comments. Three workarounds, ranked:

1. **Don't ship `config.json`; ship `config.json.example` instead.** A non-loaded sibling file with realistic commented entries. Users copy → rename to start. This keeps `loadConfig` in `src/config.ts:81-88` simple (still falls back to `DEFAULT_CONFIG` when no `config.json` exists), and gives the user a worked example. Cost: 1 file in `config-templates/`, ~25 lines.

2. **Ship `config.json` with real entries plus a `_comments` key.** Hacky but readable: `{"_comments": ["Add tools here, see PERMISSIONS.md"], "allowedTools": [...]}`. Requires `loadConfig` to ignore unknown keys (it already does — spread-merge at `src/config.ts:87`). Downside: the `_comments` key is dead weight in every project's config and not a standard JSON convention.

3. **Switch config to JSON5/JSONC.** Most user-friendly but pulls in a dependency or 30+ lines of stripper code, conflicts with the project's "no external runtime deps" rule (project CLAUDE.md), and breaks any tooling that JSON-parses `config.json`. Don't recommend.

**My recommendation: option 1 (`config.json.example`).** Aligns with how the same file shipped today as `CLAUDE.md.sample` works (`init.ts:136-146`). Could also include the proposed `Bash(npm install *)` and `mcp__<server>` placeholders so the user just needs to substitute their server name.

## Risks / Uncertainty

- **Empirical re-test gap.** I did not install an MCP server inside this worktree and re-run the failing tool against the current claude CLI to live-confirm `mcp__*` is broken. The conclusion rests on (a) the official Anthropic docs not listing it, (b) an Anthropic contributor closing #3107 saying "wildcards not supported", and (c) the user's actual reported failure on `mcp__plugin_context7_context7__query-docs` despite `mcp__*` being in their config. That's strong but not proof-of-the-current-binary. A 5-minute IMPLEMENT-time check (install context7 plugin, dispatch a worker with `allowedTools: ["mcp__*"]` only, observe block) would close this gap.
- **Possible historical drift.** It's plausible `mcp__*` worked in some older claude-code version and was tightened. I did not check version history. Doesn't change the recommendation (the docs are the contract), but would explain why the default was added in the first place.
- **Plugin-prefixed server names.** `plugin_context7_context7` looks autogenerated by claude-code's plugin system. If the prefix changes between claude-code versions, any concrete `mcp__plugin_context7_context7` rule a user writes today could silently stop matching tomorrow. Worth a callout in `PERMISSIONS.md`. No fix at the dangeresque layer.
- **`acceptEdits` permission-mode caveat.** Per Anthropic docs, `acceptEdits` only auto-approves file edits + common filesystem commands; it does NOT auto-approve MCP or arbitrary Bash. Many users (including me before re-reading the docs today) likely assume `acceptEdits` is broader than it is. Worth one sentence in `PERMISSIONS.md`.
- **Out of scope.** Did not look at codex-engine MCP permissioning. Codex configures MCP via `~/.codex/config.toml` per README.md:454-457; same UX gap likely exists but a separate report.
