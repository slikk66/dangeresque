## Design Notes

**Why a separate `src/allow.ts` instead of folding into `cli.ts`.** The behaviors (config read/write, idempotent merge, indent preservation, MCP list parse) are non-trivial and have distinct unit-testable seams. Mirrors how `stage.ts`, `init.ts`, `artifact.ts` are factored. Net new module, no parallel code path — `loadConfig`/`DEFAULT_CONFIG` semantics in `src/config.ts` are unchanged; `allow` writes raw JSON the same way the user would by hand.

**JSON formatting preservation.** `detectIndent` reads the first `\n<spaces>"` in the original raw text; falls back to 2-space if absent or tabs are involved. Trailing newline is preserved iff the original file ended in `\n`, otherwise omitted. New files created by `allow` default to 2-space indent + trailing newline (matches the project's existing `.dangeresque/config.json` shape).

**`claude mcp list` parsing.** Defensive layered approach: try `--json` first; on non-zero exit fall back to `claude mcp list` (text). The text parser tolerates `name:` prefix lines, bare-name lines, "No MCP servers" sentinel, and "Checking..." headers. JSON parser handles array-of-strings, array-of-{name}, `{servers: [...]}` envelope, and bare object map (server-name keys). All 8 shapes are covered by `parseMcpListOutput` unit tests using fabricated inputs (no live `claude` shell-out in tests).

**Why dropping `mcp__*` from default is safe.** The INVESTIGATE run proved this entry was inert under current claude-code (anthropics/claude-code#3107 closed-as-wontfix; Anthropic permissions docs do not list it). Removing it cannot regress users whose MCP tools were working — those users either had specific `mcp__<server>` rules already, or their tools were silently blocking and prompting (which fails in headless `-p`, not silently). Net effect on broken users: zero. Net effect on confused users: a clear "tool was blocked" with `dangeresque allow mcp` as the documented one-line fix.

## Risks / Uncertainty

- **End-to-end CLI smoke not run.** This worker doesn't have `node` in its allowedTools, so I could not execute `node dist/cli.js --help` or `node dist/cli.js allow bash "ls" --dry-run` to confirm the wired dispatch + arg parsing in a live process. The unit tests cover the underlying module thoroughly; the CLI layer is a thin parser on top whose shape mirrors existing commands (`cmdStage`, `cmdInit`). Recommend a VERIFY-mode follow-up that runs the CLI binary with `--help`, `allow --help` (if added later), and at least one `allow` invocation in a tmp dir.
- **`claude mcp list` shape assumption.** I could not run `claude mcp list` from the sandbox to capture its actual output format. The parser is defensive across four JSON shapes and a few text shapes, but if claude's text format on the user's machine differs (e.g. tabular columns with headers), one-line servers may parse correctly while header rows may slip into the result as garbage entries. `discoverMcpServers` deliberately throws when the binary is missing so that case fails loudly rather than producing zero entries. Mitigation: a user can always pass an explicit `<server>` to bypass discovery.
- **`dangeresque allow mcp` discovery on a machine with zero servers.** Returns `{added: [], skipped: []}` and the CLI prints `No MCP servers reported by 'claude mcp list'. Nothing to add.` Exits 0. Documented behavior — not an error.
- **Existing project users who run `dangeresque init` again.** `init.ts:68` only copies template files when the destination doesn't exist — so `PERMISSIONS.md` is created on first init but won't appear in already-initialized projects unless they run `dangeresque init` again with the new release. The init log line in `Next steps` is shown every run and links to it; users who never re-init get the README pointer instead.

## Observations (not in scope to fix)

- `loadConfig` at `src/config.ts:81` does a shallow spread-merge so a user's `allowedTools` override fully replaces the default array (no deep-merge). This is correct and matches `allow`'s assumption (always reads/writes the full array).
- `validateSetup` does not check `allowedTools` shape — a user who hand-edits `config.json` and writes `"allowedTools": "Read"` (string instead of array) would fail at runtime inside the worker, not at validate-time. Could be hardened in a follow-up.
- The legacy `claude-code#3107` is closed-as-wontfix at the upstream level; if a future claude-code version adds bare-`mcp__*` support, dangeresque's removal here would be revisitable but not urgent — the explicit `mcp__<server>` form would still work.
