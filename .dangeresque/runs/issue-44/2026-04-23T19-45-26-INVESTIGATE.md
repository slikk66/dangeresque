## Summary

Hypothesis **CONFIRMED**. `mergeWorktree` Phase 3 (`src/worktree.ts:388`) uses `git branch -d`. Because worktrees are created from `origin/main` (`src/runner.ts:522` — `git worktree add -b <branch> <path> origin/main`), git's `branch.autoSetupMerge` default auto-configures the new branch's upstream tracking to `origin/main`. After a local `dangeresque merge` the merge commit is on HEAD but the upstream tracking ref (`refs/remotes/origin/main`) is stale until the human pushes — so `-d`'s upstream-merged safety check refuses to delete, even though the branch is demonstrably in HEAD's ancestry. The proposed fix (Option A: switch to `-D`) is correct because Phase 1's `headBefore !== headAfter` check (`src/worktree.ts:348-357`) already guarantees the merge landed on local HEAD — which is the ACTUAL invariant `mergeWorktree` needs. The upstream-tracking safety is orthogonal and wrong for this workflow.

### 4. Git's documented semantics for `-d` vs `-D`

From `git branch --help`:
> **-d, --delete**: Delete a branch. The branch must be fully merged in its upstream branch, or in HEAD if no upstream was set with --track or --set-upstream-to.
>
> **-D**: Shortcut for --delete --force.

So `-d`'s "merged" check is against the upstream tracking ref (when one exists), not HEAD. `-D` skips the check entirely.

## Why #37's INVESTIGATE was wrong to preserve `-d`

`.dangeresque/runs/issue-37/2026-04-22T04-40-06-INVESTIGATE.md:177` wrote:

> `mergeWorktree` uses `git branch -d` (safe delete), `discardWorktree` uses `git branch -D` (force delete). This is the correct distinction (merge wants to refuse if somehow unmerged; discard wants to nuke regardless) and should be preserved.

That reasoning treats `-d`'s check as generic "is this branch safe to delete". But the ACTUAL semantics are "is this branch merged into its upstream ref". Those are only equivalent when upstream tracks a ref that will contain the merge — which, in the dangeresque merge-then-push workflow, is never true at `mergeWorktree` time. The real safety `mergeWorktree` needs (did the merge land on local HEAD?) is already enforced by the Phase 1 HEAD check. Preserving `-d` was over-conservative AND semantically misaligned with the workflow.
