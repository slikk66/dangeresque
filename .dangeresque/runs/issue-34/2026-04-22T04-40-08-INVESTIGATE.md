## Summary

Hypothesis in issue #34 is **correct**. `validateSetup` (src/config.ts:95-123) inspects config dir + engine NAME + prompt files only; it never asks "is the engine binary reachable?" or "is the engine authenticated?". Both failure modes described in the issue are real, with one twist per engine and one important caveat for Claude.

The two failure modes produce *different* downstream symptoms, both bad:

1. **Binary not on PATH** → `spawn("claude"|"codex", args)` at src/runner.ts:510/567/615/662 emits an `error` event with `code: "ENOENT"`; the `child.on("error", …)` handlers at 535-539/579-582/642-645/673-676 reject the `runWorker` promise with `Failed to start claude: spawn claude ENOENT`. The rejection bubbles out of `await runWorker(…)` in `cmdRun` (src/cli.ts:327, un-try/catch'd) up to the top-level `main().catch(…)` at src/cli.ts:924-927. Result: the worktree created at src/runner.ts:488 is **stranded**, no FAIL comment is posted, user just sees `Fatal: Failed to start claude: …`. Running `dangeresque run` again on the same mode+issue then hits the `Worktree already exists` guard at src/runner.ts:440-448 and double-fails.

2. **Binary present, auth missing/expired** → spawn succeeds, engine subprocess runs, prints an auth error to stdout/stderr, exits non-zero. `runWorker` resolves with `{exitCode: ≠0}`; `cmdRun` takes the failure branch at src/cli.ts:358-396, prints the `!!! DANGERESQUE RUN FAILED !!!` banner, posts a **misleading FAIL comment** on the GitHub issue (src/cli.ts:371-391) implying the worker tried and failed when it never had a chance, and exits. Worktree + branch + empty artifact dir are all left behind.

The AFK flavor makes both especially bad: the user walks away expecting ~20 min of work, comes back to a stranded worktree and (in case 2) a bogus FAIL comment attributing the failure to the model rather than to setup.

## Key Findings

### `validateSetup` is the only gate, and it's only called from `cmdRun`

Confirmed by grep — `validateSetup` appears in exactly two places: its definition at src/config.ts:95 and its single call site at src/cli.ts:187. Good news for scoping the fix: adding a stricter check there (or wiring `validateEngineRuntime(engine)` next to it, gated to `cmdRun`) automatically satisfies the "only runs on `run`" criterion — `logs`, `results`, `status`, `merge`, `discard`, `clean`, `stats`, `init`, `stage` never invoke it.

### Engine binary resolution: `spawn()` uses PATH, so ENOENT is the signal

Every spawn call in src/runner.ts uses the bare name `"claude"` or `"codex"` with no `shell:true` and no absolute path. Node's `child_process.spawn` resolves these via `$PATH` (execvp semantics) and emits an `error` event with `code: "ENOENT"` on lookup miss. A pre-flight PATH check can therefore be a pure-Node `spawnSync(engine, ["--version"], { stdio: "pipe" })` — check `result.error?.code === "ENOENT"` — with no shell dependency and no portability risk.

Alternative portable-on-POSIX-only approach: `execSync(\`command -v \${engine}\`, { stdio: "pipe" })` and catch non-zero. Either works; the `spawnSync` approach matches existing codebase style (runner.ts already uses `spawnSync` at line 741).

### Engine auth: **asymmetric reliability**

This is the substantive correction to the issue's "Proposed checks" section.

**Codex — `~/.codex/auth.json`:**
- **EXISTS** on this host (`/Users/dbilleci/.codex/auth.json` confirmed via Glob).
- This is the canonical codex auth cache location on all platforms.
- **File-check is reliable**: `existsSync(join(homedir(), ".codex", "auth.json"))` is a good proxy for "codex is logged in".
- Caveat: file being present does not *prove* the token is unexpired. Stale tokens would still fail at subprocess time. But the issue's stated goal is "avoid silent failure on obviously-unauthenticated runs" — covering the common "user never logged in" and "user logged out" cases is enough. Stale tokens are a narrower edge case and the current FAIL banner + comment is fine for those.

**Claude — `~/.claude/.credentials.json`:**
- **DOES NOT EXIST** on this host (Glob returned `No files found`).
- Confirmed I'm authenticated (this Claude Code session is running right now), yet the file is absent.
- Root cause: Claude Code on darwin stores OAuth credentials in the **macOS Keychain**, not in a dotfile. The `~/.claude/.credentials.json` file is the Linux fallback. See `config.ts:8` referencing `~/.claude/projects/` for session transcripts — this is NOT the creds path.
- **Naïve adoption of the issue's suggestion would false-negative on every darwin user**, blocking `dangeresque run` for users who are actually fully authenticated. That's worse than the current behavior.
- Reliable options for Claude auth pre-flight, with tradeoffs:
  - **(a) Skip Claude auth check entirely.** Only check binary presence. The existing post-spawn FAIL path already handles auth failure loudly (though on the issue-comment side). Simplest, zero cost, zero false-negatives.
  - **(b) Platform-conditional file check.** `process.platform === "linux"` → check `~/.claude/.credentials.json`; darwin/win32 → skip. Closes the Linux gap but leaves the dominant darwin case uncovered.
  - **(c) Darwin Keychain probe.** `spawnSync("security", ["find-generic-password", "-s", "Claude Code-credentials"], { stdio: "pipe" })` returns 0 iff a credential is stored. Reliable, zero API cost, but Keychain-item name is undocumented-API territory and could change.
  - **(d) Real auth probe.** Run `claude -p "ok"` with `--model claude-haiku-…` and a low output limit. Accurate but costs a call and adds latency on every `dangeresque run`. Overkill for the stated bug.
- **Recommendation for IMPLEMENT**: option (a) for Claude (binary-only), option (file-check) for Codex. This catches the concrete bug filed in the issue ("what happens if user runs this without being logged in to codex?") without creating a new false-negative bug for darwin Claude users.
- If Linux Claude auth coverage matters later, (b) can be added as a tightly-scoped follow-up; don't block this issue on it.

### Where to wire the check

Cleanest shape, matching existing `ValidationResult` conventions (src/config.ts:90-93):

```ts
// src/config.ts
export function validateEngineRuntime(engine: Engine): ValidationResult {
  const errors: string[] = [];

  const probe = spawnSync(engine, ["--version"], { stdio: "pipe", encoding: "utf-8" });
  if (probe.error && (probe.error as NodeJS.ErrnoException).code === "ENOENT") {
    errors.push(
      `Engine '${engine}' not found on PATH.\n` +
      `    Install it and re-run, or switch engine in .dangeresque/config.json.`
    );
    return { valid: false, errors };  // no point auth-checking if binary missing
  }

  if (engine === "codex") {
    const authPath = join(homedir(), ".codex", "auth.json");
    if (!existsSync(authPath)) {
      errors.push(
        `Engine 'codex' is on PATH but not authenticated.\n` +
        `    Run: codex login\n` +
        `    Then retry.`
      );
    }
  }
  // Intentionally no file check for claude on darwin — it uses Keychain.
  // Binary-on-PATH is the only pre-flight signal we have that won't false-negative.

  return { valid: errors.length === 0, errors };
}
```

And in src/cli.ts:187-195, add a second validation block right after the existing one, before any worktree-creating work:

```ts
const runtimeValidation = validateEngineRuntime(config.engine);
if (!runtimeValidation.valid) {
  console.error("Setup validation failed:");
  for (const err of runtimeValidation.errors) {
    console.error(`  - ${err}`);
  }
  process.exit(1);
}
```

Note: the engine must come from the fully-resolved config (after `DANGERESQUE_ENGINE` env override at src/cli.ts:198-201 and `--engine` flag at src/cli.ts:221-227). So the runtime check must go **after** CLI parsing, not beside the existing `validateSetup` call which runs before overrides are applied. Current placement at line 187 would miss `--engine codex` overrides — that's a gotcha for the implementer.
