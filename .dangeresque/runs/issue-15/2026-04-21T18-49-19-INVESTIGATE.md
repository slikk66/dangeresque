## Summary

Hypothesis in the issue is correct. `ArtifactBuilder.startedAtMs` is initialized to `Date.now()` at **construction time** via an inline field initializer (`src/artifact.ts:89`). In `cmdRun` (`src/cli.ts:264`), the builder is constructed **after** `await runWorker(...)` returns — so the captured timestamp reflects when the worker finished, not when the run started. This produces two linked symptoms in the output artifact:

1. `started_at` > `worker.started_at` (should be `<=`).
2. `duration_ms` ≈ finalization time only (seconds), not the total end-to-end run time (minutes).

Both are fixed by letting `cmdRun` supply the run-start timestamp explicitly, and having `ArtifactBuilder` fall back to `Date.now()` only when the caller doesn't provide one.

## Root Cause

`ArtifactBuilder` defaults `startedAtMs` to construction time and offers no way for the caller to supply the true run-start timestamp. `cmdRun` does capture a candidate timestamp one line before `await runWorker(...)` (`workerStartedAtMs`, line 254) — but never threads it into the builder. Nothing is structurally wrong with when the builder is constructed; the missing piece is an explicit input for run-start on `BuilderInit`.

## Proposed Minimal-Diff Plan (for IMPLEMENT)

### `src/artifact.ts`

1. Add optional field to `BuilderInit` (after line 85):
   ```ts
   /** Epoch-ms timestamp for when the overall run started. Falls back to construction time. */
   startedAtMs?: number;
   ```
2. Convert the inline `startedAtMs` initializer (line 89) to be assigned in the constructor so it can consume `init.startedAtMs`:
   ```ts
   private readonly startedAtMs: number;
   constructor(init: BuilderInit) {
     this.init = init;
     this.startedAtMs = init.startedAtMs ?? Date.now();
     this.recordEvent("run_started", { ... });
   }
   ```
   `readonly` still allows assignment inside the constructor, so the invariant is preserved.

### `src/cli.ts`

1. Capture `const runStartedAtMs = Date.now();` at the very top of `cmdRun` (recommended: right after `const projectRoot = resolveProjectRoot();` on line 135). Placing it there — rather than at line 253 — makes `duration_ms` cover arg parsing + issue fetch + worker + (optional) review + finalize, i.e. the true end-to-end run duration.
2. Pass it through to the builder:
   ```ts
   const builder = new ArtifactBuilder({
     projectRoot,
     issueNumber,
     ...(fixtureUsed ? { issueUrl: null } : {}),
     mode: effectiveMode,
     engine: config.engine,
     model: config.model,
     effort: config.effort,
     worktreeName: workerResult.worktreeName,
     branch: workerResult.branch,
     archivePath: workerResult.archivePath,
     startedAtMs: runStartedAtMs,
   });
   ```
   `workerStartedAtMs` at line 254 remains unchanged — it's the worker's own phase start.

No other call sites need updating (see Usage audit above).
