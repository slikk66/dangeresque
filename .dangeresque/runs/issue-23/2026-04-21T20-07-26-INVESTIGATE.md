## CHALLENGE Points (Implementation Tensions)

### CHALLENGE 1 — `extractMode` not exported
File: `src/worktree.ts:237`.
The issue Goal mentions "worktree/branch naming" tests, which implies `extractMode` + `extractIssueNumber`. `extractIssueNumber` is exported; `extractMode` is not.
**Recommendation:** Export `extractMode`. Single-word additive change (`function` → `export function`), zero behavior impact, zero call-site impact. Low risk.

### CHALLENGE 2 — `resolveBranch` spawns git subprocess
File: `src/worktree.ts:120`.
The issue Goal lists "resolveBranch, branch naming" under worktree tests. But `resolveBranch` calls `execSync("git rev-parse ...")` three times. The issue's own Verification Criteria says: "Coverage of pure logic only — do NOT test subprocess spawning (that's smoke-test territory)."
**Recommendation:** Drop `resolveBranch` from the unit scope. Cover `extractIssueNumber` + `extractMode` + `parseSummaryBlock` + `formatRunOneLiner` instead. All four are pure and cover the "naming" intent. Note this trade in the IMPLEMENT run result.

### CHALLENGE 3 — `formatIssueComments` not exported
File: `src/runner.ts:153`.
The issue Goal explicitly calls out "comment filtering" (verbatim). `formatIssueComments` is the only comment-filtering function and is module-private.
**Recommendation:** Export `formatIssueComments`. Additive; no call-site impact. The issue's "Do NOT refactor production code to enable testing unless strictly necessary" clause applies — exporting IS strictly necessary here because the function is explicitly listed in the Goal. Document the one-word change as a deliberate concession.

### CHALLENGE 4 — tsconfig structure vs. pinned `bin`/`main`/`types`
The `bin: "./dist/cli.js"` path assumes `rootDir: "./src"` so `src/cli.ts` → `dist/cli.js`. If we add `test/**/*` to the main `include`, TS auto-infers a new rootDir of `.`, pushing cli.js to `dist/src/cli.js` and breaking `bin`.
**Recommendation:** Use a **separate** `tsconfig.test.json` that extends the main one. Don't touch the main tsconfig.

### CHALLENGE 5 — Test output location vs. gitignore policy
`.gitignore` is off-limits per CLAUDE.md. But the test compilation needs an output directory. Options:
- **(a)** Output to `dist-test/` — ends up untracked (noisy `git status`) unless gitignored. Forbidden to gitignore.
- **(b)** Output to `dist/<subdir>/` (e.g., `dist/__tests__/`) — inherits the existing `dist/` gitignore. Clean, no config change. Tradeoff: npm `files: ["dist"]` would ship tests — not a concern for an internally-linked CLI, but worth noting. 
- **(c)** Add `dist-test/` to gitignore — forbidden by CLAUDE.md.

**Recommendation:** Option (b) — `outDir: "./dist/__tests__"`. Stays within the already-gitignored `dist/` tree. No policy violation.

## Risks / Uncertainty

1. **`node --test` with directory argument in Node 22.20.0** — should recursively discover `*.test.js`. If it fails to pick up files, fallback is a glob: `node --test 'dist/__tests__/**/*.test.js'`.
2. **Temp dir cleanup** — if a test throws before cleanup, temp dirs leak. `afterEach` handles this; test authors should not rely on `beforeEach` state across tests.
3. **`ArtifactBuilder.build()` calls `execSync("git config --get remote.origin.url")`** through `buildIssueUrl`. In a test environment inside the worktree, this will succeed and return the real remote URL. Tests for `issue_url` should either pass `issueUrl: null` override in `BuilderInit` (explicit null is honored) or tolerate whatever the live remote returns. Recommend passing `issueUrl: null` in tests to avoid environment coupling.
4. **Publish surface growth** — `dist/__tests__/` will end up in the npm tarball via `files: ["dist"]`. Not a functional issue since this is an internally-linked CLI. If future publish is planned, revisit via a separate issue (add `.npmignore` or tighten `files`).
