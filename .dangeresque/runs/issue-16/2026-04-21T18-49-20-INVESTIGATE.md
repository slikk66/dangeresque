## Summary

Issue #16 asks for `reviewModel` / `reviewEffort` config + CLI overrides with worker-value fallback. The current code has a single `model`/`effort` pair that both phases consume. Adding the override is straightforward in `src/config.ts`, `src/runner.ts`, and `src/cli.ts`. The issue's "comment header shows review model/effort when they differ" and "works for both engines" criteria force both gaps to be addressed incidentally.

## Proposed Implementation Outline (for IMPLEMENT run)

**config.ts** — add optional fields, no default entries:
```ts
export interface DangeresqueConfig {
  // ... existing ...
  /** Model for review pass (falls back to `model` when unset) */
  reviewModel?: string;
  /** Effort level for review pass (falls back to `effort` when unset) */
  reviewEffort?: string;
}
// DEFAULT_CONFIG: no additions — absence means fallback
```

**runner.ts**:
- In `buildClaudeReviewArgs`, compute `const reviewModel = config.reviewModel ?? config.model;` / `const reviewEffort = config.reviewEffort ?? config.effort;`. Use these in `--model` / `--effort` pushes.
- In `buildCodexReviewArgs`, same fallbacks on CLI model arg; append `\n\nEffort preference: ${reviewEffort} ...` to the prompt string (mirrors worker).
- Extend `CommentOptions` with `reviewModel?` / `reviewEffort?`.
- In `buildRunTag`, after emitting `model=X effort=Y`, if `opts.reviewModel && opts.reviewModel !== opts.model` add `review-model=<X>`; same for effort. Only emit when they differ.

**cli.ts**:
- Add `--review-model <name>` → `config.reviewModel = args[++i]` and `--review-effort <level>` → `config.reviewEffort = args[++i]` in the arg loop.
- Startup banner: compute effective review model/effort; if either differs, print a second line.
- Both `postRunComment` call sites: pass `engine`, `model`, `effort`, `reviewModel`, `reviewEffort` (effort conditional on engine remains as today; apply the same rule to reviewEffort).
- Update `usageForEngine` to document the two new flags.

## Risks / Uncertainty

- **Success-path comment scope creep**: today the success `postRunComment` call does not pass engine/model/effort. To make "review model/effort visible when differs" actually show up, we have to start passing them. That changes the success tag from `**[dangeresque IMPLEMENT]**` to `**[dangeresque IMPLEMENT · engine=claude · model=claude-opus-4-7 · effort=max]**` on every run (not just when review differs). Worth a comment in the implementation PR; arguably a welcome improvement for parity with failure tag.
- **Codex effort-in-review was always silent**: fixing this as a side effect of #16 is defensible (the verification criteria explicitly demand it), but it is a small behavior change for existing codex users — their review phase will now receive an `Effort preference:` line. Matches worker behavior so it is a coherent change.
- **Artifact schema lacks review fields**: if review-model/effort start mattering for evaluation, the artifact JSON should capture them. Out of scope for #16 per likely-files. File a follow-up issue if analytics on review-model diversity is desired. Would require `ARTIFACT_SCHEMA_VERSION` bump.
- **No test framework** exists (CLAUDE.md forbids adding one without authorization). Verification for implementer will rely on `yarn build` + manual invocation / fixture runs (e.g. `--issue-fixture`) to confirm CLI flags plumb through.
- **`DEFAULT_CONFIG` intent**: issue says "leave out of `DEFAULT_CONFIG` so 'unset' is the default". `loadConfig` uses `{ ...DEFAULT_CONFIG, ...raw }`, so merely omitting the keys from DEFAULT_CONFIG is sufficient — `config.reviewModel` will be `undefined` unless the user sets it. Nullish-coalescing in the builders handles the rest.
- **CLI flag name collision**: verified — neither `--review-model` nor `--review-effort` is currently used (grep confirmed).

## Observations

- `--effort` is ignored in codex mode today (cli.ts line 233 warns). The mirror warning for `--review-effort` in codex would now be misleading because with this change, codex review WILL honor effort (via prompt injection). So the warning logic around `effortFlagUsed` and `--review-effort` + codex needs the implementer's judgment: either (a) drop the warning entirely now that codex honors effort through the prompt, or (b) leave the worker warning alone and do NOT warn on `--review-effort` in codex (since review does honor it once this change lands). Recommend (b) as the minimal change — don't touch the existing worker warning semantics.
- `buildRunTag` tag format uses ` · ` separator. Sticking to the same separator for `review-model=X` keeps the tag readable.
- The codex review builder does not use `--effort` CLI flag (codex exec doesn't expose one) — that's why effort must go into the prompt. Consistent with how worker handles it.
