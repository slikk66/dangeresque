## Design Decisions

### Why pass `config.reviewModel` (raw) instead of the resolved `effectiveReviewModel` to `postRunComment`?

Both were considered. Passing `effectiveReviewModel` (already-resolved to worker value when unset) forces the tag's "differ" check to always compare effective-to-worker, which is natural for claude. But for codex, the existing code passes `effort: undefined` to hide effort from the tag. If we also pass `reviewEffort: effectiveReviewEffort` (say "max"), the comparison `"max" !== undefined` is true → tag shows `review-effort=max` even when user did nothing. Misleading.

Passing `config.reviewEffort` (undefined unless explicitly set) sidesteps this: the tag only shows `review-effort=` when the user actually overrode it AND the override differs from the worker value. For claude this also works correctly — `config.reviewEffort === undefined` when unset → tag skips. When set and different → tag includes.

This matches the user-facing semantics: "the tag shows review-X when the user intentionally made the review different."

### Why add `engine`/`model`/`effort` to the success-path `postRunComment`?

Previously the success path passed only minimal fields, so the tag read `**[dangeresque IMPLEMENT]**`. The failure path already passed the full set, producing `**[dangeresque IMPLEMENT · engine=claude · model=claude-opus-4-7 · effort=max]**`. Issue #16 verification criterion "GitHub comment header shows review model/effort when different from worker" requires the worker baseline to be present in the tag for the "different from" relation to make sense. Extending the success path to match the failure path is the minimal scope expansion that makes this criterion observable. No other behavior changes.

### Why inject `Effort preference:` into codex review prompt?

The codex worker (`buildCodexWorkerArgs` line 319) already injects `Effort preference: ${config.effort} ...` because `codex exec` has no `--effort` CLI flag. The codex review builder previously skipped this. After this change, `--review-effort` has no effect in codex mode without adding the injection. Since the issue requires "Works for both engines" AND "Same fallback semantics for `reviewEffort`", the injection is forced. Matches worker pattern so it's a coherent change.
