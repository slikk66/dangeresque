# IMPLEMENT — Interactive worktree picker (issue #46)

## Status

`implemented, unverified`

## Summary

Implemented the interactive numbered picker per the staged IMPLEMENT plan and the INVESTIGATE artifact. Four command handlers (`cmdLogs`, `cmdResults`, `cmdMerge`, `cmdDiscard`) now fall through to a readline-based picker when the branch arg is omitted AND stdin is a TTY. Piped/non-TTY invocations preserve today's error-on-missing behavior. Existing arg-form invocations are unchanged.

## Files Changed

1. **`src/picker.ts` (NEW, 45 lines)** — `pickWorktree(worktrees, opts?)` with injectable `input`, `output`, `isTTY`, `label`. Returns `undefined` in all failure/cancel paths so callers fall through uniformly. Defaults: `input = process.stdin`, `output = process.stderr` (never pollutes stdout), `isTTY = (input as NodeJS.ReadStream).isTTY === true`.

2. **`src/worktree.ts`** — added `WorktreeFilter = "all" | "running" | "finished"` type + `filterWorktrees()` pure function.

3. **`src/cli.ts`** — added `resolvePositionalOrPick()` helper; wired into `cmdLogs` / `cmdResults` / `cmdMerge` / `cmdDiscard`. Per-command filter matches INVESTIGATE's recommendation:
   - `cmdLogs` → `"all"` (IDLE worktrees still have transcripts)
   - `cmdResults` → `"all"`
   - `cmdMerge` → `"finished"` (can't merge a branch being written to)
   - `cmdDiscard` → `"finished"` (arg-form still bypasses for power users)
   - `cmdResults` / `cmdMerge` / `cmdDiscard` became `async`; `cmdMerge` / `cmdDiscard` signatures changed from `(branch: string | undefined)` → `async (args: string[])`. Main dispatch updated with `await` + `args.slice(1)`.

4. **`test/unit/picker.test.ts` (NEW, 78 lines)** — 5 cases using `PassThrough` + injected `isTTY`:
   - non-TTY + non-empty list → `undefined`, no output written
   - TTY + valid "2\n" → returns `worktrees[1]`
   - TTY + out-of-range "99\n" → `undefined`
   - TTY + non-numeric "abc\n" → `undefined`
   - TTY + empty list → `undefined`, no prompt

5. **`test/unit/worktree.test.ts`** — added 3 `filterWorktrees` cases (`all`, `running`, `finished`).

## Observations

- `readline` was already imported elsewhere (`src/logs.ts:3`), so stdlib usage has precedent.
- `formatMissingTargetError` is reused unchanged for `cmdLogs`/`cmdResults` no-picker fallback. `cmdMerge`/`cmdDiscard` keep their simpler `Usage: …` message to avoid widening scope (INVESTIGATE §253 flagged consistency as out-of-scope).
- `process.stderr` is the picker's default output stream — prompts never pollute stdout piping.
- `--immutable` install was required (sandbox allowlist); worktree had no `node_modules` at entry.

## Risks / Uncertainty

1. **cmdMerge / cmdDiscard signature change**: internal only, not re-exported from `src/index.ts`. Low breakage risk.
2. **Terminal echo**: relies on shell canonical mode. Works in bash/zsh/iTerm/Terminal/tmux. Exotic CI pseudo-TTYs with canonical disabled would lose echo — acceptable per INVESTIGATE §263.
3. **`isTTY === true` strict check**: avoids truthy-on-odd-type false positives.
4. **Ctrl-C cancel UX**: default SIGINT → exit 130. No fancy cancel messaging added, per INVESTIGATE §138 "Out of scope — Fancy Ctrl-C message".
5. **Live smoke test unverified**: the exact no-arg-in-real-terminal flow was not driven from this worker (no TTY inside the sandbox). Unit tests cover the picker logic via `PassThrough` streams with explicit `isTTY` injection, which exercises the same code path.
