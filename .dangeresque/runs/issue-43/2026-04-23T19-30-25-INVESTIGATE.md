# Issue #43 — claude engine leaks task prompt in argv

## Status
investigating

## Summary

Hypothesis is **confirmed**. The claude worker and review paths assemble `buildTaskPrompt(opts, archivePath)` / `buildReviewPrompt(...)` (which contain the GitHub Issue body + staged comments + archive path + diff context) and push them as the final positional argv to the `claude` CLI. Node's `child_process.spawn` forwards those args as argv to the claude binary, where `ps -p <pid> -o command=` or `ps -ef` exposes the full task content to any process on the host.

A clean fix exists with **no new runtime dependencies and no temp files**: `claude -p` reads the user prompt from stdin when no positional prompt is supplied. Dangeresque currently spawns claude with `stdio: "inherit"` (parent TTY), so stdin is the user's terminal. Switching to `stdio: ["pipe", "inherit", "inherit"]` in headless mode lets the parent write the prompt to `child.stdin` and close it, while preserving live stdout/stderr on the parent TTY.

The shape of the fix is a near-mirror of the #35 codex fix — strong case for folding both into one IMPLEMENT PR.

Because `args[args.length-1]` is the full task prompt (issue body + filtered comments + archive path; ~5–30KB typical), every byte lands in argv and is visible via `ps` to any local process on the host. `stdio: "inherit"` attaches the child's stdin directly to the parent TTY, so even if claude tried to read stdin, it would get whatever the parent terminal is doing (not a prompt we control).

What's already safe in the claude path (confirmed):
- `workerPrompt` / `reviewPrompt` template files are passed via `--append-system-prompt-file <path>` — only the path shows in argv, not file contents. This is what the issue says and it checks out.
- `--model`, `--effort`, `--permission-mode`, `--session-id`, `--name`, etc. are all short strings safe to expose.
- `--allowed-tools` / `--disallowed-tools` patterns are static config strings — not sensitive.

The leak is strictly the dynamic task prompt at runner.ts:332 and :387.

## Investigation Questions

### 1. Does the claude CLI support reading the user prompt from stdin?

**Yes.** Source: the official CLI reference docs at https://code.claude.com/docs/en/cli-reference explicitly document the piping pattern:

> `cat file | claude -p "query"` — Process piped content — `cat logs.txt | claude -p "explain"`

And the headless/Agent-SDK page at https://code.claude.com/docs/en/headless shows this exact shape with **zero positional prompt**:

```bash
gh pr diff "$1" | claude -p \
  --append-system-prompt "You are a security engineer. Review for vulnerabilities." \
  --output-format json
```

No positional prompt appears in that command. The only way for claude to have a user message to respond to is for the stdin content to become that user message. So `claude -p` reads the user prompt from stdin whenever a positional arg is omitted (and `--input-format` is `text`, which is the default).

Also relevant flags surfaced from docs + local `claude --help` (v2.1.118):

| flag | behavior | relevance |
|---|---|---|
| `-p`, `--print` | non-interactive, "useful for pipes" | already used when `config.headless=true` |
| `--input-format <text\|stream-json>` | default `text`; stream-json reads JSON msgs from stdin | stick with default `text` for dangeresque |
| `--output-format <text\|json\|stream-json>` | output shape | unchanged |
| `positional prompt` | "Your prompt" — optional | **omit it; feed via stdin instead** |
| `--system-prompt` / `--system-prompt-file` | replaces default system prompt | wrong for task prompt (see Q2) |
| `--append-system-prompt` / `--append-system-prompt-file` | appends to system prompt | already used for worker-prompt.md / review-prompt.md; wrong for task prompt (see Q2) |

Behavior summary: with `-p` and `--input-format=text` (the default), claude reads stdin as the user's turn when no positional prompt is provided. When both positional AND stdin are provided (e.g. `cat file | claude -p "explain"`), the positional is the prompt and stdin is attached as additional user-turn content — we DON'T want that combined form because it still leaks the positional via argv. The solution is omit-positional.

### 2. Does the claude CLI support a `--prompt-file` or equivalent for the USER prompt?

**No.** `claude --help` (filtered for `file|stdin|input|prompt`) and the full CLI reference show only four file-backed flags and they're all for SYSTEM prompt:

| flag | target |
|---|---|
| `--system-prompt <string>` | replace default system prompt (string, leaks via argv) |
| `--system-prompt-file <path>` | replace default system prompt from file (path-only, no argv leak) |
| `--append-system-prompt <string>` | append to default system prompt (string, leaks) |
| `--append-system-prompt-file <path>` | append to default system prompt from file (path-only, no argv leak) |

Using these flags for the task prompt is **semantically wrong** and would change behavior:
- System prompt sets the model's instructions/persona. In headless mode without a user prompt, claude's turn loop has nothing to respond to. Empirically the expected outcome is an empty-input error, or an assistant turn without a user message — either way, a behavior change from the current `args.push(buildTaskPrompt(…))` user-turn flow.
- The existing `--append-system-prompt-file` slot is already taken by `worker-prompt.md` / `review-prompt.md`. Multi-append usage isn't documented beyond the bracket form `--append-system-prompt[-file]`.
- Even if it worked, it conflates "worker rules" (system) with "your specific issue" (user) — hostile to context-cache reuse across runs and a regression vs. the current shape.

Stdin is the only correct non-argv channel for the USER prompt.

### 3. Temp-file fallback needed?

**No.** Stdin is strictly better than any tempfile approach for the same reasons listed in #35:

- No cleanup lifecycle; prompt lives only in-process memory bridging node → claude.
- No filesystem-readable leak (another local user could read a temp file with permissive perms; defeats the whole point).
- No new state in `.dangeresque/` hygiene, no new `.gitignore` entries, no new cleanup paths on crash.
- The verification criteria in the issue say "No temp files written to disk (or clearly cleaned up if used)" — stdin sidesteps the question entirely.

Not recommended.

### 4. What about non-headless (interactive) mode?

When `config.headless === false`, dangeresque drops the `-p` flag and spawns claude as an interactive session. In that mode stdin is a TTY (the user's terminal) used for typing messages. Piping the initial task prompt over stdin would either:
- Fail because claude's interactive UI won't accept a pre-pipe (docs show stdin-piping only for `-p`).
- Or accept it but leave stdin closed afterward — breaking the ability to continue interactively.

Conclusion: the stdin fix applies only when `headless === true`. For `headless === false` we must keep the positional-arg form. This is an acceptable tradeoff because:
- Default config is `headless: true` (src/config.ts:53). All AFK runs use it.
- Non-headless is for human-interactive debugging; the operator is already looking at their own TTY and would see the prompt anyway.
- Documented as a known limitation in the fix commit message.

No work-around exists without upstream claude CLI support for a `--prompt-file` flag. If users want non-headless argv hygiene, file an issue upstream.

## Proposed Fix (for IMPLEMENT mode — fold with #35)

Scope: `src/runner.ts` only. Two build functions + two spawn sites. Near-mirror of the #35 codex fix.

### 1. Refactor `buildClaudeWorkerArgs` (runner.ts:299-335)

Return shape goes from `{ args: string[]; workerSessionId: string }` to `{ args: string[]; workerSessionId: string; prompt: string }`.

```ts
// OLD
args.push(buildTaskPrompt(opts, archivePath));
return { args, workerSessionId };

// NEW
const prompt = buildTaskPrompt(opts, archivePath);
return { args, workerSessionId, prompt };
```

Args list ends with `--session-id <uuid>`. No positional prompt appended. `headless` gating for `-p` stays as-is (line 306-308).

### 2. Refactor `buildClaudeReviewArgs` (runner.ts:337-390)

Same shape: return `{ args: string[]; reviewSessionId: string; prompt: string }`. Move the positional `buildReviewPrompt(...)` out of args and into the return value.

### 3. Worker spawn (`runWorker` claude branch, runner.ts:627-655)

```ts
const { args, workerSessionId, prompt } = buildClaudeWorkerArgs(opts, worktreeName, archivePath);
// ...
const useStdin = opts.config.headless;
const child = spawn("claude", args, {
  cwd: opts.projectRoot,
  stdio: useStdin ? ["pipe", "inherit", "inherit"] : "inherit",
  env: { ...process.env },
});

if (useStdin) {
  child.stdin?.on("error", (err) => {
    // EPIPE if claude exits before consuming stdin — downstream close handler covers it
    console.warn(`dangeresque: claude stdin write failed: ${err.message}`);
  });
  child.stdin?.write(prompt);
  child.stdin?.end();
}
```

Note: when `headless === false`, keep the old `"inherit"` behavior AND append the prompt as the positional arg (since there's no other way). The refactor is still worth it — IMPLEMENT can centralize that fallback in one place:

```ts
if (!useStdin) args.push(prompt);  // interactive-mode fallback, leaks via argv (documented)
```

### 4. Review spawn (`runReview` claude branch, runner.ts:722-743)

Identical changes. Same `useStdin` gate on the same `config.headless`.

## Risks / Uncertainty

- **EOF signaling.** Claude `-p` with `--input-format=text` reads stdin to EOF. Forgetting `child.stdin.end()` after `write` would hang the worker indefinitely. Mitigation: call `end()` in the same tick as `write()`.
- **EPIPE on stdin.write.** If claude exits early (e.g. auth failure, invalid flag), the parent's `write()` can throw EPIPE. Mitigation: attach an `error` listener on `child.stdin` that logs and proceeds; the `child.on("close", ...)` exit-code path already handles downstream failure.
- **Backpressure.** Prompt size is bounded: issue body (a few KB), up to 3 recent comments + all staged comments (a few KB each), and fixed boilerplate. Real-world runs are in the 5–30KB range. Node's default stream highWaterMark is 16KB; a single `write(prompt)` call may return `false` if over, but Node buffers internally. `end()` flushes when the child drains. No action needed for realistic sizes.
- **Not fully verified end-to-end.** I did not spawn a claude process under the proposed change in this run (INVESTIGATE mode). The stdin/pipe behavior is inferred from official docs + `claude --help`. The Agent SDK / headless page explicitly shows the pattern `gh pr diff "$1" | claude -p --append-system-prompt "..." --output-format json` with no positional prompt, which is exactly what we'd do. A live claude run with `ps` inspection should be the final proof under VERIFY.
- **Non-headless argv leak remains.** When `headless: false`, the positional prompt is still appended (no alternative without upstream CLI support). Default config is headless so this doesn't affect AFK-worker security posture; document it in the commit message and/or a README note under IMPLEMENT. Not a regression (current behavior is the leak; the fix is strictly an improvement for the headless case).
- **Claude CLI version skew.** Stdin-as-prompt via `claude -p` is old documented behavior and should work on v2.1.118 (the locally installed version) without issue. If a future claude CLI ever removes stdin-piping, our fix would break loudly with a usage error — no silent failure mode.
- **No live test of the no-positional case.** The docs pattern `gh pr diff | claude -p --append-system-prompt "..." --output-format json` is the closest published evidence that stdin-only works. I couldn't run `echo "hi" | claude -p` locally because the sandbox blocks multi-op shell. IMPLEMENT / VERIFY should run one live smoke test.
