# Issue #43 — IMPLEMENT: claude engine argv prompt leak fix

## Status
implemented, unverified

## Summary

Mirrors the codex fix shipped in #35 (b94b24c) onto the claude engine.

Two claude build functions (`buildClaudeWorkerArgs`, `buildClaudeReviewArgs`) now return a third field `prompt: string` alongside the existing `args` + session id. The dynamic task prompt (issue body + filtered comments + archive path) is no longer pushed positionally onto `args` under headless mode — instead, the caller writes it to `child.stdin` and closes stdin in the spawn site. `stdio` flips from `"inherit"` to `["pipe", "inherit", "inherit"]` under headless so stdout/stderr remain live on the parent TTY while stdin is a controlled pipe.

When `config.headless === false` (interactive operator mode), the prompt is still appended positionally — there is no way to pre-pipe when stdin is the operator's TTY. This is the documented fallback and matches the INVESTIGATE recommendation.

Both functions are now `export function` (were file-private) so unit tests can import them. Four new unit tests cover argv hygiene on both modes for both worker and review.

No runtime dependencies added. Codex paths untouched.

   - `buildClaudeWorkerArgs` → `export function`, returns `{ args, workerSessionId, prompt }`. Non-headless path pushes `prompt` positionally as fallback.

## Risks / Uncertainty

- **Non-headless argv leak is intentional and documented.** `headless=false` still pushes the prompt positionally because there is no way to pipe stdin when it is the operator's TTY. Default config is `headless=true` so AFK runs benefit; operator-interactive runs see their own prompt on their own ps, acceptable tradeoff per INVESTIGATE §122-133.
- **EOF signaling.** `child.stdin?.end(prompt)` (single call) both writes and closes in the same tick, following the codex fix pattern. If the write buffer is huge and end() returned before full drain, Node buffers internally until the child reads — realistic prompt sizes (5–30 KB) stay well under the default 16 KB high-water mark and flush on the next tick.
- **PID file timing.** The existing `setTimeout(..., 3000)` for worker pid-file write is untouched — still races the worktree-create barrier. Stdin write happens synchronously after spawn, well before the 3-second pid-file window.
- **Engine-parity maintained.** Codex path (`buildCodexWorkerArgs` / `buildCodexReviewArgs`) is unchanged in this commit. Both engines now follow the same shape: build function returns `{args, prompt}`, spawn site writes prompt via stdin.
